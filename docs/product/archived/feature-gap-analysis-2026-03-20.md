# 功能差距分析：从"聊天工具"到"医生的个人AI诊所"

> 日期：2026-03-20
> 基于：代码审计 + 竞品分析（左手医生、爱达健康、华佗GPT、Freed、Nabla）

## 产品定位

**左手医生：** 医院的AI信息化基础设施 — 卖给三甲医院信息科，嵌入HIS系统
**我们：** 医生的个人AI诊所 — 卖给个体/私人诊所医生，微信即入口

核心差异：**医生拥有自己的数据和患者关系。** 不依赖医院系统，不需要IT部门。

## 二、竞品功能对照

### 与左手医生逐项对比

| 功能域 | 左手医生 | 我们 | 差距评级 |
|-------|---------|------|---------|
| **诊前：预问诊** | 成熟：图形选型+多轮问答+结构化 | 有：文字多轮+7字段，缺预览 | 小差距 |
| **诊前：智能导诊** | 有：症状→科室+医生匹配 | 不需要（单医生产品） | 不适用 |
| **诊中：听译** | 核心产品：6麦阵列+声纹分离 | 无（聊天式输入，不同赛道） | 不适用 |
| **诊中：结构化病历** | 从听译自动生成，对接HIS | 从聊天/语音/图片生成，独立存储 | 持平 |
| **诊中：辅助诊断** | 疾病预测模型+知识图谱推理 | 零实现 | **大差距** |
| **诊中：智能开方** | 基于诊断推荐处方 | 无 | 差距（暂不做） |
| **诊后：用药管理** | 60家三甲上线，服药提醒+不良反应 | 无 | 差距 |
| **诊后：随访管理** | 自动入组+交互对话+智能外呼 | 有任务系统，但只通知医生 | **中差距** |
| **诊后：患者教育** | 扫码获取个性化用药/饮食指导 | 无 | 差距 |
| **架构：知识图谱** | 6000+疾病图谱，三阶段构建 | 依赖LLM通用知识 | 差距（Phase 4） |
| **部署** | 医院内网+硬件+HIS集成 | 云端SaaS，零部署 | **我们的优势** |

### 我们独有的（左手医生没有）

| 功能 | 说明 |
|------|------|
| 医生拥有数据 | 数据属于医生，不属于医院 |
| 微信零部署 | 医生注册即用，无需IT/硬件/HIS |
| Action Chip快速操作 | 确定性快速路径，跳过LLM，更快更便宜 |
| 多输入病历录入 | 图片/PDF/语音/文字统一入口 |
| 患者消息通道 | 患者直接联系医生（虽然目前是单向的） |

## 三、差距修复优先级

### P0 — 补完闭环（已有功能的最后一公里）

已有代码 80%，差最后 20% 无法形成闭环。投入小、价值大。

#### P0-1：QR码患者入口

**现状：** 患者注册接口已有（`POST /api/patient/register`），但没有QR码入口。

**需要做：**
- `GET /api/doctor/{doctor_id}/qrcode` — 生成带 doctor_id 的二维码图片
- 二维码指向 `https://{domain}/patient/register?doctor_id={id}`
- 医生工作台设置页增加"我的二维码"展示+下载

**工作量：** 0.5天
**文件：** `channels/web/chat.py`（新endpoint）、`frontend/web/`（设置页）

#### P0-2：患者预问诊确认预览

**现状：** 预问诊采访完成后直接提交，患者看不到生成的摘要。

**需要做：**
- 采访完成后进入 `reviewing` 状态，展示结构化摘要给患者
- 患者可以"确认提交"或"修改补充"
- 确认后才生成病历+通知医生

**工作量：** 1天
**文件：** `domain/patients/interview_session.py`、`channels/web/patient_interview_routes.py`、患者端前端

#### P0-3：患者通知能力

**现状：** `send_doctor_notification()` 完整可用，但没有对应的 `send_patient_notification()`。

**需要做：**
- `send_patient_notification(patient_id, message)` — 通过微信模板消息推送
- 任务到期时同时通知医生和患者
- 预问诊提交后通知患者"医生已查看"
- 患者消息回复后通知患者

**工作量：** 2天
**文件：** `domain/tasks/notifications.py`（新函数）、`domain/tasks/scheduler.py`（触发双向通知）

**依赖：** 微信小程序模板消息权限（需要患者关注/授权）

#### P0-4：医生→患者回复通道

**现状：** 患者发消息，医生收到通知，但没有回复接口。

**需要做：**
- `POST /api/doctor/reply` — 医生回复患者消息
- 回复存入 `patient_messages` 表（已有）
- 触发 `send_patient_notification()` 推送给患者
- 医生工作台增加"患者消息"入口（可作为新的Action Chip）

**工作量：** 1.5天
**文件：** `channels/web/chat.py`（新endpoint）、`frontend/web/`（消息入口）

---

### P1 — 核心差异化

#### P1-1：AI辅助诊断

**现状：** `Action.diagnosis` 枚举存在，`_dispatch_action_hint` 返回 `None`（fall through）。14字段结构化数据已有。

**需要做：**
- 诊断提示词设计（输入：14字段结构化病历 → 输出：初步诊断2-3个 + 鉴别诊断3-5个 + 推荐检查）
- `_dispatch_action_hint` 实现 `Action.diagnosis` handler
- 前端展示：病历确认后自动触发诊断分析，结果显示在病历下方
- 医生可以接受/修改/拒绝诊断建议

**工作量：** 3-5天
**文件：** `agent/handle_turn.py`、新建 `domain/diagnosis/` 模块、`frontend/web/`

**技术决策：**
- Phase 1 用 LLM prompt（快速上线）
- Phase 2 考虑知识图谱增强（ADR 0022）

#### P1-2：患者端"我的健康"页面

**现状：** 患者门户只能看病历列表和发消息。

**需要做：**
- 治疗时间线视图（按时间排列：就诊→检查→复诊）
- 当前待办清单（下次复诊日期、需要做的检查）
- 当前用药清单（从病历 treatment_plan 字段提取）
- 医生信息卡片（名字、科室、头像）

**工作量：** 2-3天
**文件：** `channels/web/patient_portal.py`（新endpoint）、患者端前端

#### P1-3：医生档案增强

**现状：** doctors 表只有 name/specialty/accepting_patients。

**需要做：**
- 数据库增加字段：`avatar_url`、`bio`、`clinic_name`、`clinic_address`、`consultation_hours`
- 医生设置页增加档案编辑
- 患者端展示医生卡片（头像+名字+科室+简介）

**工作量：** 1天
**文件：** `db/models/doctor.py`、`channels/web/`（设置页）、患者端

## 四、实施路线

```
第1周：P0（补完闭环）
  ├── P0-1 QR码入口               0.5天
  ├── P0-2 预问诊确认预览           1天
  ├── P0-3 患者通知能力             2天
  └── P0-4 医生→患者回复            1.5天
                                   ─────
                                   5天

第2-3周：P1（核心差异化）
  ├── P1-1 AI辅助诊断              3-5天
  ├── P1-2 患者端"我的健康"          2-3天
  └── P1-3 医生档案增强             1天
                                   ─────
                                   6-9天

第4周+：P2（增强体验）
  └── 按需推进
```

**里程碑：**
- P0 完成 → 第一个医生可以完整走通"扫码→预问诊→看病→随访"全流程
- P1 完成 → 与左手医生在核心功能上持平，且有私人诊所差异化优势
- P2 持续 → 建立数据飞轮，拉开竞争距离

## 五、不做的事（及原因）

| 功能 | 不做原因 |
|------|---------|
| 智能开方 | 处方合规风险高（药品数据库、医保对接、法律责任）。等诊断功能验证后再考虑 |
| HIS/EHR对接 | 目标用户（私人诊所）大多没有HIS。独立运行是优势 |
| 环境听译/硬件 | 与"微信即诊所"轻量定位矛盾。聊天式输入够用 |
| 智能导诊 | 单医生产品不需要科室分诊 |
| 全科知识图谱 | 规模太大（"ocean"）。先用LLM prompt，Phase 4再考虑神经科专科图谱 |
