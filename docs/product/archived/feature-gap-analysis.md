# 功能差距分析与实施路线图

> 更新时间：2026-03-08｜基于竞品研究 + 代码库深度审查 + 双 Agent 评审

---

## 一、核心矛盾：最大差异化功能在日常路径中不工作

产品把「专科量表」定位为核心护城河，但 NIHSS 只能在批量导入管道里触发。当医生在微信里说「患者卒中，NIHSS 6 分，较上次改善」，系统把它扔进自由文本，一个数字都不会结构化提取。**产品在宣传上承诺的专科差异化，在最核心的使用场景里完全缺席。** mRS 和 UPDRS 则在整个系统中根本不存在。

## 三、三个阻断性缺口

不修复就无法留存第一批用户。

---

### 缺口 1 — 专科量表不在聊天流程中 ❌

**战略判断：** 这是第一周就会导致医生放弃的原因。

#### 现状

- `NeuroCaseDB` 有 `nihss` 字段，但只有 `/api/neuro/` 批量导入管道才能写入
- 常规聊天流程（`structure_medical_record()`）不提取任何专科评分
- mRS、UPDRS、MMSE、MoCA 在整个系统中完全不存在

#### 实现方案（中等复杂度）

**新建数据表 `specialty_scores`：**

```
id              INT PK
record_id       INT FK → medical_records.id
doctor_id       VARCHAR FK → doctors.doctor_id
patient_id      INT FK → patients.id
score_type      VARCHAR   # NIHSS | mRS | UPDRS | MMSE | MoCA | seizure_log | GCS
score_value     FLOAT
raw_text        TEXT      # 原始提取片段（用于可溯源审核）
details_json    TEXT      # 评分细项，如 NIHSS 各子项
source          VARCHAR   # chat | import | manual
extracted_at    DATETIME
```

**在 `services/ai/structuring.py` 中插入后处理步骤：**

```python
async def structure_medical_record(text, ...):
    record = await _existing_structuring_logic(text)

    # 新增：专科评分后处理
    if _has_specialty_keywords(text):   # fast check，不额外调 LLM
        scores = await extract_specialty_scores(text)
        record.specialty_scores = scores

    return record
```

**`fast_router` 新增量表关键词检测：** 只有含 NIHSS / mRS / UPDRS / MMSE 等词时才触发额外 LLM 调用，常规接诊不增加延迟。

**实现顺序：**
1. NIHSS + mRS（复用已有神经科逻辑，最快）
2. MMSE / MoCA
3. UPDRS + 癫痫发作日志（结构更复杂）

---

### 缺口 2 — 初诊与复诊没有区分 ❌

**战略判断：** 不影响第一次试用，但会阻断第二周留存。没有复诊模板的系统，在医生眼里等同于「不懂我的工作」。

#### 现状

所有病历走同一个 `add_medical_record` 流程，输出相同的 8 字段结构，无论是新患者首诊还是第 10 次复诊。复诊时不会自动引用上次诊断和用药。

#### 实现方案（小复杂度）

**`medical_records` 表新增 2 个字段：**

```sql
ALTER TABLE medical_records ADD COLUMN encounter_type VARCHAR(32) DEFAULT 'unknown';
-- first_visit | follow_up | unknown
```

**新建 `detect_encounter_type()` 函数：**

```python
async def detect_encounter_type(session, doctor_id, patient_id, text):
    records = await get_records_for_patient(session, doctor_id, patient_id)
    if not records:
        return "first_visit"
    if any(kw in text for kw in ["复诊", "随访", "复查", "回诊"]):
        return "follow_up"
    return "unknown"
```

**两套结构化 Prompt：**

- **首诊模板**：强调完整采集主诉、现病史、既往史、家族史、过敏史、基线量表评分
- **复诊模板**：自动带入上次诊断和用药作为上下文，输出格式聚焦「本次变化」——症状变化、用药依从性、方案调整

**数据库迁移：** 现有记录中每位患者最早一条标为 `first_visit`，其余标为 `follow_up`。

---

### 缺口 3 — 没有合规病历 PDF 导出 ❌

**战略判断（由第 4 位提升至第 3 位）：** 没有 PDF，医生必须另外维护一套正式记录系统，产品永远是「草稿辅助工具」而非「核心工作系统」。这是产品完整性的最后一公里。

#### 现状

结构化病历存储在数据库，Web 端可查看，完全没有导出功能。

#### 实现方案

**MVP 简化版（优先做，<100 行代码）：**

- `weasyprint`：Markdown → PDF，满足基本打印需求
- 页眉：「AI 辅助生成草稿——执业医师审核签名后方为正式病历」
- 新建 `/api/export/record/{id}/pdf` 端点
- Web 患者时间轴新增「导出」按钮

**正式版（后续迭代）：**

- `reportlab` 生成符合卫生部 2010 格式的 A4 病历
- 结构：患者信息表 → 主诉 → 现病史 → 既往史 → 体检 → 专科量表 → 诊断 → 医嘱 → 医师签名区
- 所有导出操作记入审计日志

## 四、重要但非阻断性缺口

---

### 缺口 4 — 专科指标纵向趋势图 ⚠️（依赖缺口 1）

**战略价值：** 神经科患者的 NIHSS / mRS 随时间变化趋势，是纵向患者关系的核心可视化。无竞品提供此能力。

#### 实现方案（小-中复杂度）

**后端：** 新建 `/api/patient/{id}/scores-timeline?score_type=NIHSS` 时间序列 API

**前端：**

```bash
npm install recharts
```

新建 `ScoresTrendChart` 组件，嵌入患者详情页：

- 下拉切换 NIHSS / mRS / UPDRS / MMSE / MoCA
- 点击数据点回显原始文本（溯源）
- 趋势评估：NIHSS 改善 ≥4 分有临床意义；触发警告文案

---

### 缺口 5 — 诊断感知随访计划自动生成 ⚠️（依赖缺口 1）

**战略价值：** 当前关键词风险分级是粗糙的代替品，不体现专科价值。真正的价值是：AI 识别出「卒中患者，mRS 2 分，已调整抗板治疗」，自动建议「1 个月后随访，复查血压和凝血」。

#### 实现方案（小-中复杂度）

**新建 `followup_rules.py` 规则库：**

| 诊断 | 基础间隔 | 风险因素修正 |
|---|---|---|
| 脑卒中 | 14 天 | NIHSS≥15 → 3 天；溶栓后 → 3 天 |
| 帕金森 | 60 天 | UPDRS≥50 → 30 天；运动波动 → 2× |
| 癫痫 | 30 天 | 突破发作 → 7 天；难治性 → 14 天 |
| 高血压 | 30 天 | 合并糖尿病肾病 → 14 天 |
| 心衰 | 14 天 | 急性心衰 → 3 天 |

**病历保存后异步触发 `create_auto_followup_task()`**，输出：
- 推荐随访间隔 + 日期
- 监测项目清单（如「血压、症状评估、用药依从性」）
- 危险信号清单（如「血压>180/110，新发神经症状」）
- 推荐理由文本

---

### 缺口 6 — 患者端微信问卷（单独立项，最低优先）

**战略价值：** 解决「20% 卒中患者 3 个月内失访」问题。但需要独立的患者端渠道，是单独的工程模块。

#### 实现方案（中等复杂度）

**数据模型：**
- `questionnaire_templates`：问卷模板（系统预设 + 医生自建）
- `questionnaire_instances`：向患者分发的实例（含 UUID token）
- `questionnaire_responses`：患者回答

**患者访问：** 无需登录，通过 Token 链接 `/patient/questionnaire/{token}` 填写

**前端：** 新建 `PatientQuestionnairePage.jsx`，支持单选/多选/量表/文本题型

**提醒：** APScheduler 每日检查未提交问卷，通过企业微信发送提醒

**患者回答：** 自动归档为随访记录，关联至患者时间轴

## 五、实施路线图

```
第一阶段（2-3 周）——首付款前必须完成
  ┌─ 1. 专科量表提取（聊天流程）
  │     → NIHSS + mRS（两个核心字段，复用已有神经科逻辑）
  │     → SpecialtyScore 表 + 后处理 LLM
  │     → fast_router 量表关键词检测（不影响常规接诊延迟）
  │
  └─ 2. 初诊/复诊模板分离
        → encounter_type 字段 + 检测逻辑（关键词为主）
        → 两套结构化 Prompt
        → 数据库迁移（现有记录自动标注）

第二阶段（2-3 周）——提升留存
  ┌─ 3. PDF 病历导出（MVP 简化版优先）
  │     → Markdown→PDF（weasyprint，先跑通闭环）
  │     → 逐步升级为卫生部 2010 格式
  │
  └─ 4. 专科指标趋势图
        → 依赖第一阶段 SpecialtyScore 数据
        → Recharts + 时间序列 API
        → 临床意义评估（NIHSS 改善≥4 分有意义）

第三阶段（3-4 周）——完整产品
  ┌─ 5. 诊断感知随访计划
  │     → 规则库（卒中/帕金森/癫痫/高血压/心衰）
  │     → 自动任务生成 + 监测项目/危险信号文案
  │
  └─ 6. 患者端微信问卷（独立立项）
        → 问卷模板 + Token 访问 + 患者填写 UI
        → APScheduler 提醒 + 自动归档
```

## 七、被低估的风险

### 风险 1：AI 幻觉在神经科场景是灾难级的

把 NIHSS 6 分写成 9 分、把「无溶栓适应症」写成「建议溶栓」——医生签字后出事，医生背锅。Abridge 把 97% 幻觉检测率作为核心技术差异化对外宣传不是偶然，这是医生采用 AI 病历工具的最大心理障碍。

**当前处理：** 确认门存在，但 UI 上没有显式的「此为 AI 草稿，请仔细核对」提示，高风险字段（量表评分、诊断）没有置信度标注。

**建议：** 确认弹窗中对专科量表字段加粗标红提示核对；评分提取时附带 `raw_text` 原文供医生对照；这应该是**优先级 2**，不是优先级 6。

### 风险 2：没有任何真实目标用户访谈数据

所有战略判断建立在二手研究（多中心学术文献、峰会报告、产品页面）和代码分析上。三甲医院神经科医生每半天接诊 45 人的数据，不等于私立诊所医生的真实痛点。

**建议：** 在第一阶段开发的同时，平行访谈 3-5 名自由执业神经内科医生。用他们的反馈校准优先级，而不是继续依赖间接数据。

### 风险 3：「自由执业医生」是更精准的靶向

在多家医院轮转、患者记录分散在多套 HIS、用私人微信管理私下患者的自由执业神经科副主任医师——对他们来说，产品是真正的「个人病历系统」，痛点最尖锐，付费意愿最强。当前所有文档把目标客户描述为「私立诊所医生」，但这个人群的具体化版本——自由执业医生——才应该是主要叙事和首批目标用户。

### 风险 4：商业化路径完全空白

目标客单价是多少？直销还是口碑传播？首批用户怎么找？没有这些答案，功能做完了也没有医生来用，也无从判断产品是否有商业可行性。

### 风险 5：企业微信合规灰区被低估

私立诊所医生个人开通企业微信 KF 的资质路径需要明确。2025 年监管收紧「限制医生用社交媒体身份将患者引流至非官方渠道」，与产品核心使用场景（医生通过微信接收患者病情描述）有潜在摩擦，需要法律论证而非一句「合规定位」带过。

## 九、一句话结论

> **方向完全正确，但核心差异化功能在最重要的使用路径中不工作。正确的下一步是停止横向扩展，用 4-6 周集中交付专科量表采集 + 初诊复诊模板 + PDF 导出这三件事，然后找 3-5 名自由执业神经内科医生深度试用，用真实反馈决定后续方向。**
