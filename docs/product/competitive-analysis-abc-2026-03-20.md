# 竞品深度对比：ABC诊所管家 vs. doctor-ai-agent

> 日期：2026-03-20
> 基于：ABC官网功能页、产品拆解报告、DeepSeek集成系列博客、Codex战略审查反馈

## 背景

ABC诊所管家（abcyun.cn）是基层医疗SaaS头部产品，3万+诊所客户覆盖300+城市，
95个城市医保对接。创始团队来自腾讯/微软，有医学背景。2023年深度集成DeepSeek大模型。

Codex 战略审查指出：我们的真正竞品不是左手医生（医院级）或美国ambient scribe，
而是面向中小诊所的SaaS/CRM工具。ABC诊所管家是最直接的对标。

## 三、核心发现

### 1. ABC功能全面碾压，但方向不同

ABC是一个**完整的诊所操作系统**：病历+处方+药品+收费+医保+营销+供应链。
我们不可能也不应该复制这个广度——ABC花了多年+多轮融资才到今天。

### 2. 我们唯一且关键的领先点：AI预问诊

ABC的AI是**医生端辅助工具**：医生输入症状 → AI推荐诊断。
我们的AI是**患者端自助入口**：患者扫码 → AI采访 → 医生收到现成病历。

这是根本性的流程差异：

```
ABC的流程：
  患者到诊所 → 医生问诊（15-20分钟）→ 医生用AI写病历 → 开处方

我们的流程：
  患者扫码（家里）→ AI采访（10分钟）→ 医生收到病历 → 确认（2分钟）→ 诊断
```

**ABC节省的是"医生写病历的时间"。我们节省的是"医生问诊的时间"。**
后者价值更大：一个私人诊所医生一天多看3-5个患者 = 直接增加收入。

### 3. ABC的弱点

| ABC弱点 | 原因 | 我们的机会 |
|---------|------|----------|
| **无患者端AI入口** | ABC定位是"诊所管理"不是"医患连接" | 我们做ABC做不到的事 |
| **诊前是空白** | ABC从患者到诊所才开始介入 | 我们从患者扫码就开始 |
| **过于全面导致复杂** | 进销存、医保、供应链增加学习成本 | 我们只做临床核心，零学习成本 |
| **不做专科深度** | 全业态覆盖=每个专科都不够深 | 我们从神经科做深再扩展 |
| **供应链绑定** | 商业模式依赖药品采购提成 | 我们纯软件，不绑定利益冲突 |

### 4. 正确的市场策略

**不与ABC正面竞争，做ABC的"上游"：**

| ABC解决的 | 我们应该解决的 |
|----------|-------------|
| 医生在诊所里的效率 | 患者到达诊所之前的准备 |
| 诊所内部管理 | 医患之间的连接 |
| AI帮医生做决策 | AI帮患者表达自己 |
| 诊中+诊后管理 | 诊前+诊后沟通 |

甚至可以考虑：**与ABC等诊所SaaS通过API对接**，而不是竞争。
我们做患者端获客+预问诊+随访沟通，他们做诊所内管理+处方+收费。

## 四、战略启示：修正后的产品优先级

结合ABC对比 + Codex审查反馈，修正优先级：

### 必须做（核心差异化）

| 项目 | 理由 | 优先级 |
|------|------|-------|
| **完善预问诊闭环**（QR码+患者预览+通知） | 这是我们唯一领先ABC的点，必须做到极致 | **P0** |
| **AI辅助诊断** | ABC已有98%准确率的诊断，我们不能在诊断上完全空白 | **P0**（降低范围：先做初步诊断建议，不做完整的证型/处方推荐） |
| **患者端体验**（"我的健康"页面） | ABC没有患者端，这是差异化 | **P1** |

### 应该做（补短板）

| 项目 | 理由 | 优先级 |
|------|------|-------|
| **双向消息通道** | ABC有在线咨询，我们只有单向消息 | **P1** |
| **患者通知** | ABC有微信随访推送，我们患者零通知 | **P1** |
| **医生档案** | 患者端需要看到医生信息 | **P1** |

### 不做（ABC已经做得很好且我们没有优势的）

| 项目 | 理由 |
|------|------|
| 药品进销存 | ABC核心壁垒+供应链绑定，我们无法竞争 |
| 收费/财务 | 需要医保对接，复杂度极高 |
| 电子处方+审方 | 合规风险高，ABC有20万+药品知识图谱 |
| 病历模板库 | ABC有几百种模板，我们LLM生成每次即用 |
| 会员营销 | ABC+沃享/智晓慧已经很成熟 |
| 连锁管理 | 目标用户是单体诊所 |

### 长期考虑（P2+）

| 项目 | 理由 |
|------|------|
| **与诊所SaaS集成** | API对接ABC/领健等系统，共享病历数据 |
| **专科知识图谱** | 从神经科开始，做比ABC更深的专科AI |
| **患者社区/教育** | ABC没有，且患者粘性强 |

## 五、产品定位总结

```
ABC诊所管家 = 诊所的ERP（管理一切）
我们         = 医生的AI前台（帮医生获客、接诊、跟踪）

ABC从"诊所管理"出发，向"AI辅助"扩展。
我们从"AI临床"出发，向"患者连接"扩展。

不同的起点，不同的护城河。
```

**我们的一句话定位：**
> 在患者到达诊所之前，AI已经完成了问诊；在患者离开诊所之后，AI继续跟踪随访。
> ABC管好诊所内的一切，我们管好诊所外的一切。

## 六、风险与未回答的问题

1. **ABC是否会补上患者端AI预问诊？** 很可能——他们有技术能力和客户基础。
   我们的窗口期有限，必须在预问诊上快速建立口碑和数据壁垒。

2. **"诊所外"能独立成为产品吗？** 还是必须和"诊所内"绑定才有价值？
   需要验证：医生是否愿意用两个系统（ABC管诊所+我们管患者）。

3. **私人诊所医生的真实付费意愿？** ABC通过供应链（药品采购提成）变相降低
   SaaS费用。纯软件收费能否被接受？

4. **专科深度 vs. 全科广度？** ABC覆盖全业态但每个都不深。我们从神经科深入
   是否真的构成壁垒？还是市场太小？

5. **合规风险：** AI诊断建议的法律责任边界未定义。ABC声称98%准确率但也没有
   公开的临床验证数据。我们需要明确免责声明和审计留痕。

## 七、Codex 战略审查反馈（2026-03-20）

> 以下为 OpenAI Codex 对本文档的独立审查意见，原文保留，未经编辑。

### 事实纠正

- **ABC 客户规模：** 本文档写"3万+"，ABC当前官网写的是**10万+**。基础事实需对齐。
- **"ABC诊前是空白"不成立：** ABC有微信预约、续方、随访、微信触达。准确表述应为
  "ABC缺少患者端AI多轮预问诊"，而不是"无患者端"。
- **"多输入领先"不稳：** ABC也支持语音/图片/PDF输入。我们最多是"统一入口更灵活"，
  不是明确领先。

### 战略盲点

1. **真正的威胁不是功能抄袭，是渠道碾压。** ABC已有10万+安装量、实施团队、培训、
   续费关系。他们补一个患者端AI预问诊比我们拿下一家诊所容易得多。

2. **忽略了分发问题。** 中国私立诊所买单常常先看获客、收费、医保、续方、药房、合规，
   不先看"AI预问诊"。预问诊不是购买决策的第一驱动力。

3. **忽略了使用链条。** 没有预约、提醒、通知、病历回写、微信触达、医生工作台闭环，
   预问诊只是一个孤立问卷。

4. **忽略了"谁录入谁负责"。** 很多医生不会信患者自填病史，助理会重新问一遍。
   如果医生不信任预问诊结果，核心价值直接塌掉。

5. **10分钟AI采访可能是流失点，不是优势。** 患者完成率未经验证。

### 市场定位批评

- **"管理诊所外的一切"太空，像广告不像可售卖产品。**
- 作为 standalone 产品，这个定位很弱。诊所外的事天然依赖诊所内系统（预约、病历、
  收费、复诊、处方、通知）。
- 如果不能回写现有系统，本质上是在制造第二套工作流。医生不会为此单独维护一套系统。
- **更合理的定位不是"诊所外的一切"，而是"现有诊所系统上的患者接诊与随访层"。
  也就是附着，而不是平行。**

### 优先级批评

- **AI辅助诊断放P0是方向错误。** 最缺的不是"再做一个ABC已经在卖的高风险能力"，
  而是接入、闭环、留存、证明ROI。
- **真正的P0应该是：** 扫码入口、患者完成率、医生确认流、通知触达、病历回写/导出、
  基础集成、效果埋点。
- **双向消息和患者通知应进P0，不是P1。** 没有它们就没有复诊、随访、患者回流。
- **"与诊所SaaS集成"放P2太晚。** 如果不是系统 of record，集成是生死线，不是远期优化。

### 致命风险

| 风险 | 严重程度 | 说明 |
|------|---------|------|
| **双录入杀死adoption** | 致命 | 没有集成=医生要在两个系统录入同一份病历 |
| **ABC两月内补平差异** | 高 | "微信表单+AI总结"是ABC能快速上线的轻功能 |
| **监管合规** | 高 | 患者健康数据、诊断建议、病历生成、微信触达均涉合规 |
| **一次错误分诊** | 高 | 漏问红旗症状、误导患者，医生立刻停用 |
| **神经外科私诊市场太窄** | 中 | ICP风险——流程非标、患者教育成本高 |
| **纯软件收费模式** | 中 | 基层医疗"系统+供应链抽成"比纯SaaS更好卖 |
| **组织能力** | 中 | 不只是做AI，还要做微信运营、实施交付、培训、客户成功 |

### 缺乏数据支撑的声称

| 声称 | 问题 |
|------|------|
| "一天多看3-5个患者" | 无数据，未经验证 |
| "零学习成本" | 大概率不真实 |
| "供应链绑定是弱点" | 对客户可能是价格补贴和粘性来源 |
| "专科深度能形成壁垒" | 没有TAM、客单价、复诊频次数据 |
| ABC"98%准确率" | 未追问定义、样本、病种、评估方法 |
| "患者端是差异化" | 没有证明医生愿意为患者端买单 |

### Codex 底线判断

> **这份文档最大的问题是：把"一个 feature gap"写成了"战略护城河"。**
>
> 你们现在不是在"领先 ABC 某一块"，而是在"试图证明一个附属模块值得单独购买"。
>
> 如果不把集成、闭环、完成率、医生采纳率、复诊转化、合规边界放到最前面，
> 这个策略会死在采用率上，不会死在模型效果上。

## 八、修正后的行动项（基于 Codex 反馈）

### 需要立即回答的问题（在写任何代码之前）

1. **医生会信任患者自填的病史吗？** 找3-5个目标用户做访谈，拿到真实反馈。
2. **患者愿意花10分钟完成AI采访吗？** 需要完成率数据，哪怕是内部测试。
3. **医生愿意用两个系统吗？** 如果不愿意，集成就不是P2而是P0。
4. **神经外科私诊的TAM有多大？** 如果不够大，需要重新选专科或做全科。

### 修正后的 P0（基于 Codex 反馈）

| 原P0 | 修正 |
|------|------|
| QR码入口 | 保留 |
| 预问诊确认预览 | 保留，但增加**患者完成率埋点** |
| 患者通知 | 保留（从P1提升） |
| 医生→患者回复 | 保留（从P1提升） |
| **AI辅助诊断** | **降级到P1** — 先证明预问诊有人用，再做诊断 |
| **病历导出/回写** | **新增到P0** — 至少支持PDF/Word导出，让医生能把病历搬到现有系统 |
| **效果埋点** | **新增到P0** — 预问诊完成率、医生确认率、患者复诊率必须可追踪 |

### 需要重新评估的战略假设

- "诊所外"作为独立产品 vs. 作为现有SaaS的附加层
- 纯软件收费 vs. 增值服务收费
- 专科深度（神经科）vs. 全科广度
- 自建完整系统 vs. 与ABC/领健API集成
