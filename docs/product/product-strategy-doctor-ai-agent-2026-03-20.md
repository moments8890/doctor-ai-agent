# 产品战略重定位：医生的术后随访AI助手

> 日期：2026-03-20
> 基于：甲方需求确认 + Codex战略咨询（xhigh推理，320K tokens）

## 一、真实客户画像

| | 描述 |
|---|---|
| **谁** | 大医院神经外科医生 |
| **场景** | 手术/门诊后，大量私域患者通过微信持续跟进 |
| **痛点** | 下班后被200+条微信消息淹没，无法逐一回复 |
| **需求** | AI作为医生的"分身"，处理诊后患者沟通，医生只处理真正需要关注的 |

**典型患者消息：**
- "刀口有点肿正常吗"
- "这个药吃到什么时候"
- "CTA什么时候复查"
- "头痛加重了怎么办"（← 这个需要紧急上报医生）

**医生的理想状态：**
早上起来看到一个绿/黄/红队列，而不是200条未读消息。
- 绿色：AI已自动回复（常规问题）
- 黄色：AI起草了回复，等医生确认
- 红色：紧急情况，需要医生立即介入

## 二、产品定位修正

| | 之前 | 现在 |
|---|---|---|
| **定位** | 医生的个人AI诊所 / WeChat诊前采集层 | **医生的术后随访AI助手** |
| **一句话** | — | **"张医生的随访助手"：安全地帮医生处理诊后患者消息** |
| **核心价值** | 病历结构化 + 预问诊 | **消息减负：常规问题自动回复，危险信号立即上报** |
| **AI角色** | 诊所管理工具 | **医生的数字分身** — 了解每个患者的手术史、用药、恢复阶段 |
| **场景** | 诊中 → 诊前 | **诊后（术后恢复、复查跟进、用药咨询）** |

**这不是：**
- 诊所管理系统（ABC诊所管家的赛道）
- AI诊断工具（医生已经做过诊断了）
- 预问诊采集（患者已经看过了）
- 通用医疗聊天机器人（不了解患者具体情况的）

### 核心发现

**截至2026年3月20日，中国市场没有明显的主导产品在做"医生个人拥有的、微信原生的、术后随访AI助手"。**

- 平台型（好大夫/京东/支付宝）→ 医生不拥有数据和患者关系
- 医院型（讯飞/左手医生）→ 医院所有，医生换医院就没了
- 最近的竞品（杏仁医生/未来医生）→ 需要深入调研其实际功能和市场占有率

**我们的机会：医生拥有的、微信原生的、专科深度的术后随访AI。**

### 核心差异化原则：医生数据主权

所有平台型竞品（好大夫/京东/支付宝）的商业模式都建立在"平台拥有医生数据"之上。
医院型竞品（讯飞/左手医生）的数据属于医院，不属于医生。
我们做出根本性的不同承诺：

**1. 医生的数据完全归医生所有**
- 患者资料、病历记录、对话历史、随访数据 — 所有数据的所有权归医生个人
- 平台不使用医生数据训练通用模型、不向第三方共享、不用于商业分析
- 医生删除账号 = 数据彻底删除，不保留

**2. 医生的知识可以完全导入导出**
- 医生积累的专科随访Playbook、常见问答、回复模板、患者管理经验 — 随时可导出
- 支持标准格式导出（JSON/PDF/Markdown），可迁移到任何其他系统
- 从其他系统迁入同样无障碍 — 不设置迁移壁垒

**3. 所有数据属于个人，不属于平台**
- 这不只是隐私政策上的一句话，而是架构层面的设计原则
- 医生换医院、换平台、甚至停用我们的产品，数据完整带走
- 我们的商业模式建立在"工具价值"上（帮医生省时间），不建立在"数据锁定"上

**为什么竞品做不到这一点：**

| 竞品 | 数据归属 | 为什么不会改 |
|------|---------|------------|
| 好大夫/京东/支付宝 | 平台所有 | 数据是平台估值的核心资产，让医生带走=削弱自己 |
| 讯飞/左手医生 | 医院所有 | 卖给医院信息科的产品，医生不是客户 |
| 杏仁医生 | 平台所有（推测） | SaaS+平台模式，医生数据是产品壁垒 |
| **我们** | **医生所有** | **我们卖的是工具，不是数据。医生用得好自然续费，不需要锁定** |

这三条原则是我们的**品牌承诺**，也是竞品结构上无法复制的差异化 —
因为复制意味着放弃他们现有的商业模式。

## 四、核心产品设计

### Day 1 杀手功能：患者级别的诊后自动应答

不是通用医疗问答。是**基于这个患者的手术记录、出院医嘱、当前用药、恢复阶段**来回答。

```
患者问："刀口有点肿正常吗？"

通用AI回答：
  "术后伤口轻微肿胀是正常的，如果持续加重请就医。"
  （← 无用。不了解这个患者）

我们的AI回答：
  "张阿姨，您3月15日做的动脉瘤夹闭手术，现在是术后第5天，
   轻微肿胀是正常的恢复过程。如果出现以下情况请立即联系张医生：
   伤口周围明显发红/渗液/体温超过38°C。
   您的下次复查CT是3月25日，请记得按时复查。"
  （← 知道手术类型、日期、当前恢复阶段、下次复查安排）
```

**Day 1 价值：** 如果第一周不能把医生的夜间消息处理时间砍掉至少1小时，这不是产品。

### 每个患者的最小上下文

AI必须知道的：

| 字段 | 说明 | 必需 |
|------|------|------|
| 患者身份 | 姓名 + 授权联系人（家属代发消息） | 是 |
| 诊断/手术 | 类型 + 日期 + 侧别/部位 | 是 |
| 当前恢复阶段 | 术后第X天 / 恢复期 / 稳定期 | 是 |
| 出院医嘱 | 医生批准的出院指导（注意事项/活动限制/饮食） | **是（核心）** |
| 当前用药 | 药名 + 剂量 + 疗程 + 过敏史 | 是 |
| 正常 vs. 危险信号 | 当前阶段预期症状 vs. 需紧急上报的信号 | **是（安全基础）** |
| 下次复查/随访 | 时间 + 检查项目 | 是 |
| 最近医生决策 | 上次医生亲自回复的内容（保持一致性） | 推荐 |

**神经外科专科补充字段（按需）：**
- 手术亚型：动脉瘤/脑膜瘤/脊柱/外伤/分流术/癫痫
- 关键影像日期
- 当前神经功能状态
- 硬性上报规则（如：突发剧烈头痛/呕吐/意识改变必须立即上报）

### 消息分级机制

| 级别 | 触发条件 | 处理方式 | 示例 |
|------|---------|---------|------|
| **绿色** | 常规问题，出院医嘱已覆盖 | AI自动回复 | "药什么时候吃"、"可以洗头吗" |
| **黄色** | 需要专业判断但不紧急 | AI起草回复 → 等医生确认发送 | "伤口有点痒"、"想提前复查" |
| **红色** | 危险信号关键词 / 超出出院医嘱范围 | 立即通知医生 + 给患者急救指导 | "突然剧烈头痛"、"呕吐"、"抽搐"、"肢体无力" |

### 工作流

```
患者微信发消息
  ↓
系统识别患者身份 → 加载患者上下文（手术/用药/恢复阶段）
  ↓
消息分级（绿/黄/红）
  ├── 绿色：AI基于出院医嘱自动回复（标注"AI助手"）
  ├── 黄色：AI起草回复 → 推送给医生 → 医生确认/修改 → 发送
  └── 红色：立即推送给医生 + 给患者发急救提示
  ↓
医生端展示：
  - 今日绿色 32条（已处理）
  - 今日黄色 5条（待确认）
  - 今日红色 1条（紧急）
  ↓
所有对话记录存档 + 审计日志
```

## 五、风险与安全

### 安全架构：AI为主，规则兜底，医生为辅

安全机制不是"硬编码一堆关键词"，而是**四层纵深防御**：

```
第一层（主力）：AI语义判断
  ↓ LLM理解消息的真实含义和紧急程度
  ↓ "头有点晕"（正常恢复） vs "突然剧烈头痛站不住"（紧急）
  ↓ AI能理解上下文：术后第3天的轻微头痛 ≠ 术后第30天的新发头痛

第二层（安全网）：规则兜底
  ↓ 关键词/模式匹配，防止AI判断失误
  ↓ "抽搐""昏迷""大出血""呼吸困难"等 → 无论AI怎么判断，强制红色
  ↓ 这一层是保险，不是主力

第三层（边界）：出院医嘱范围
  ↓ AI只回答医嘱覆盖的问题
  ↓ 超出范围 → 自动升级为黄色（等医生确认）
  ↓ "我可以吃XX药吗"（医嘱没提到的药）→ 不自动回答，转医生

第四层（红线）：产品禁止线
  ↓ 无论AI多有信心，绝对不做：新诊断、改药、替代急救
  ↓ 这是产品设计层面的硬约束，不是AI的判断
```

**为什么是"AI为主，医生为辅"而不是"每条都要医生确认"：**
- 如果每条消息都要医生确认，医生还是要处理200条消息，产品没有价值
- 绿色消息（60%+）AI直接回复，医生完全不需要看
- 只有黄色（需要专业判断）和红色（紧急）才推给医生
- 这样医生每天从200条变成5-10条 — 这才是真正的减负

### 致命风险

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **错误安慰** | AI把恶化的症状判断为"正常恢复" | AI语义判断为主 + 关键词规则兜底。宁可过度上报（把绿色判成黄色可以，把红色判成绿色不行） |
| **错误上下文** | 搞错患者/搞错术后天数/用了过期的医嘱 | 每次回复前验证患者身份+上下文时效 |
| **患者以为是医生回复** | 信任/期望风险 + 法律风险 | **必须明确标注"AI助手回复"，不得伪装成医生** |
| **合规** | 互联网诊疗监管（2022）：AI不能替代医生做诊断/处方。AI生成内容标识办法（2025.9）：必须标注AI生成 | AI只回答出院医嘱范围内的教育/随访问题，不做新诊断/处方 |
| **数据安全** | 患者健康数据通过微信传输 | 端到端加密存储，审计日志，最小权限 |

### 安全红线（产品禁止线）

**AI绝对不能做的：**
- 新诊断（"您可能是XX病"）
- 修改用药（"您可以停药了"）
- 替代急救（"不用去医院"）
- 伪装成医生回复

**AI可以做的：**
- 解释已有的出院医嘱
- 告知正常恢复进程（基于术后天数+手术类型）
- 提醒复查/用药时间
- 收集症状更新（记录但不诊断）
- 紧急情况引导拨打120/联系医生

### 分级容错原则

**宁可误报，不可漏报：**
- AI把绿色判成黄色 → 可接受（医生多看一条消息）
- AI把黄色判成红色 → 可接受（医生多一次紧急通知）
- AI把红色判成黄色 → **不可接受但有规则兜底**
- AI把红色判成绿色 → **绝对不可接受，规则层强制拦截**

## 六、MVP定义（90天，1个医生）

### 只做这些

| 模块 | 具体内容 |
|------|---------|
| **患者入口** | 出院时给患者扫二维码 → 进入WeCom/小程序随访通道（不是私人微信bot） |
| **患者上下文录入** | 医生/助理录入：诊断+手术日期+出院医嘱+用药+下次复查。可以从现有病历导入 |
| **自动应答（绿色）** | AI基于患者上下文判断+回答常规问题。标注"AI助手"。覆盖6类：解释医嘱、用药确认、正常vs异常、症状收集、复查安排、紧急引导 |
| **医生审批（黄色）** | AI判断需要专业确认 → 起草回复 → 推送给医生 → 一键确认/修改 → 发送 |
| **紧急上报（红色）** | AI语义判断+规则兜底双重检测 → 给患者急救提示 + 立即通知医生 |
| **医生看板** | 绿/黄/红队列 + 今日处理量 + 待确认列表 |

### 不做这些

| 不做 | 原因 |
|------|------|
| AI诊断/鉴别诊断 | 安全风险高，法规限制，医生已有诊断 |
| 处方/用药建议 | 合规红线 |
| 诊前预问诊 | 不是当前痛点（患者已经看过了） |
| 收费/医保/进销存 | ABC的赛道 |
| 通用医疗问答 | 价值太低，风险太高 |

### 成功标准

| 指标 | 目标 |
|------|------|
| 常规消息自动处理率 | >60% |
| 紧急情况漏报率 | **0** |
| 医生每日节省时间 | ≥1小时 |
| 患者满意度（愿意继续使用） | >80% |

## 七、商业模式

### 付费方买家不是医院IT，是医生本人

| 阶段 | 定价 | 包含 |
|------|------|------|
| 付费试用 | ¥8,000-20,000 / 2-3个月 | 历史对话导入、医生风格调优、专科playbook、白手套实施 |
| 续费 | ¥2,999-6,999/月/医生 | 按活跃患者数或对话量分级 |
| 团队版（后期） | ¥X/月/科室 | 多个医生 + 共享患者池 |

**为什么能收这个价？**
- 1小时/天 × 30天 = 30小时/月的专家时间
- 神经外科医生的时间价值远高于¥3,000/月
- 实际是在帮医生维护私域患者关系 = 维护收入来源

**不做的变现方式：** 患者收费、药品提成、转诊佣金（利益冲突，破坏信任）

## 八、增长路径

### 第一阶段：一个医生，做到极致

- 1个神经外科医生
- 1个术后随访工作流
- 做到医生离不开

### 第二阶段：神经外科内扩展

同一专科的不同亚型：
- 动脉瘤/蛛网膜下腔出血术后
- 脑肿瘤术后
- 脊柱手术术后
- 外伤术后
- 分流术后
- 癫痫术后

每个亚型 = 不同的恢复时间线 + 不同的正常/异常信号 + 不同的随访方案

### 第三阶段：跨专科复制

选择同样痛点的专科（高焦虑恢复期 + 重复性随访问题 + 可规则化的危险信号 + 强私域患者池）：

| 专科 | 适合度 | 原因 |
|------|-------|------|
| 骨科/脊柱 | 极高 | 术后恢复周期长、功能锻炼指导多、复查频繁 |
| 泌尿外科 | 高 | 术后管道管理、复查安排 |
| 甲状腺/乳腺外科 | 高 | 术后用药调整、定期复查 |
| 心脏介入 | 高 | 术后用药严格、生活方式管理 |
| 整形外科 | 中 | 术后护理指导多，但医疗风险相对低 |

### 第四阶段：护城河

- **专科随访Playbook库** — 每个手术类型的恢复时间线、正常/异常信号、标准问答
- **患者时间线记忆** — AI越用越了解每个患者
- **医生回复风格学习** — AI越来越像这个医生本人
- **管理式随访通道** — 不是私人微信的脆弱hack，是医生品牌的专属通道

## 九、渠道决策：不能用私人微信bot

Codex 特别强调的关键决策：

**不能在医生的私人微信上做自动回复bot。** 原因：
- 微信封号风险
- 无法控制消息格式和标注
- 无审计日志
- 无法标注"AI生成"（违反2025.9法规）
- 患者无法区分是医生还是AI

**正确方案：** 出院时给患者扫码进入受控通道（WeCom客服账号/小程序/服务号）。
这个通道：

### 关键差异放大

#### 差异1：数据主权 — 我们是唯一"医生拥有数据"的产品

```
ABC诊所管家   → 数据属于诊所/平台，医生离开诊所=失去数据
未来医生       → 数据属于医联平台，医生是内容供应商
京东健康       → 数据属于京东，医生的"数字分身"归平台运营
好大夫         → 数据属于蚂蚁集团，医生离开平台=患者关系断裂
讯飞智医随行   → 数据属于医院，医生调岗/离职=从零开始
左手医生       → 数据属于医院信息科，医生没有控制权
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
我们           → 数据100%属于医生个人，随时导出，停用即删
```

**为什么这很重要：** 一个神经外科医生积累了500个术后随访患者的对话记录、
个人回复偏好、专科知识库——这些是他几十年临床经验的数字化。在其他平台上，
这些数据帮平台赚钱；在我们这里，这些数据永远属于医生。

#### 差异2：专科深度 vs. 通用广度

```
其他产品的AI：
  患者："手术后头痛正常吗？"
  AI："术后头痛很常见，建议观察，严重时就医。" ← 通用回答，什么手术都适用

我们的AI：
  患者："手术后头痛正常吗？"
  AI："王叔叔，您3月10日做的右侧大脑中动脉动脉瘤夹闭术，
       现在是术后第10天。这个阶段的轻微头痛是正常的，
       多和颅骨愈合有关。但如果出现突发剧烈头痛、呕吐、
       意识模糊，请立即拨打120并联系张医生。
       您的下次CTA复查是3月25日。" ← 知道手术类型、日期、恢复阶段、复查安排
```

**为什么竞品做不到：** 通用平台要覆盖几十个专科，无法为每个亚专科
（动脉瘤/脑膜瘤/脊柱/外伤/分流术/癫痫）定制恢复时间线和危险信号规则。
我们从一个专科做深，每个手术类型都有精确的Playbook。

#### 差异3：医生品牌 vs. 平台品牌

```
京东健康：患者看到的是"京东健康·张医生智能体" → 患者认的是京东
好大夫：  患者看到的是"好大夫在线·张医生" → 患者认的是好大夫
讯飞：    患者看到的是"XX医院智能随访" → 患者认的是医院
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
我们：    患者看到的是"张医生的术后随访助手" → 患者认的是张医生本人
```

**为什么这很重要：** 对于有私域患者池的专科医生，品牌就是信任。
患者信的是"张主任"，不是某个平台。我们帮医生强化个人品牌，
而不是把医生变成平台的流量入口。

#### 差异4：场景聚焦 — 诊后随访，不是全流程

| 产品 | 试图覆盖 | 结果 |
|------|---------|------|
| ABC | 诊前+诊中+诊后+管理+营销+供应链 | 每个都做，每个都不深 |
| 京东/好大夫 | 挂号+问诊+处方+随访+科普+电商 | 平台逻辑，医生是工具人 |
| 讯飞 | 预防+治疗+康复+随访+科研 | 要医院采购，部署周期长 |
| **我们** | **只做诊后随访沟通** | **一件事做到极致** |

医生不需要又一个"什么都能做"的平台。他需要的是：
**每天晚上200条微信消息变成3条需要他亲自看的。**

### 竞品无法复制的结构性优势总结

| 我们的优势 | 为什么竞品做不到 |
|----------|---------------|
| **数据归医生** | 平台型竞品的估值建立在数据之上，放弃=自废武功 |
| **知识可导出** | 平台靠锁定医生维持粘性，允许导出=失去壁垒 |
| **医生品牌化** | 平台需要自己的品牌露出，让医生品牌盖过平台=自相矛盾 |
| **专科术后深度** | 通用平台覆盖几十个专科，无法为每个亚专科做精确Playbook |
| **独立于任何医院/平台** | 医院型产品绑定机构，平台型绑定流量，我们只绑定医生本人 |
| **零部署/零IT** | 医院型需要信息科采购，我们医生自己注册就能用 |
