# 产品战略：专科医生的AI助手

> 日期：2026-03-20
> 基于：甲方医生需求原话 + 竞品分析 + Codex多轮战略审查

## 统一定位

> **为专科医生打造的AI助手：像医生助手一样问诊、写病历、给初步建议、做随访
> — 医生只需做最终决策。**

这就是主任医师的真实工作模式：他不会自己问完所有病史。他让住院医/研究生
先问好、写好病历、给初步建议，他来"查房"做决策。我们的AI就是这个角色。

## 一、真实客户画像

| | 描述 |
|---|---|
| **谁** | 大医院神经外科医生，有大量私域患者 |
| **他说** | "这个智能体就是取代一个初级医生的功能" |
| **痛点1** | 门诊前没有好病历：要自己从头问诊，费时费力 |
| **痛点2** | 门诊/手术后消息淹没：200+条微信问"刀口肿不肿""药吃到什么时候" |
| **痛点3** | 缺少AI辅助：其他系统有诊断建议，他想要但不知道怎么接入 |
| **需求** | AI全程辅助：问诊→病历→诊断建议→随访，医生只做最终决策 |

**医生原话摘要：**
> "如果病人医生在看到病人前就有一份相对比较好的病例，他就会很舒服"
> "我只要做一个最终的决策，这个智能体就是取代一个初级医生的功能"
> "希望能够有初步的诊断、鉴别诊断以及进一步治疗的建议，我在里面只要筛选"
> "到复诊、用药调整时会提醒双方，执行病人随访方案"
> "要有自己核心专科小模型在手才能安全"
> "他们是给医院做的，我们是给医生做的，使用量级完全不一样"

## 二、产品定位

| | 说明 |
|---|---|
| **定位** | 专科医生的AI助手 |
| **一句话** | 像医生助手一样问诊、写病历、给初步建议、做随访 — 医生只需做最终决策 |
| **AI角色** | 医生的助手 — 做完基础工作（问诊/病历/初步建议/随访），医生做决策 |
| **核心价值** | 诊前有好病历、诊中有AI建议、诊后有自动随访 |
| **起步专科** | 神经外科（症状有限，可做深度知识库，再扩展其他专科） |
| **交付方式** | 微信小程序（患者端）+ Web工作台（医生端） |

### AI助手做什么 vs. 医生做什么

| 工作 | AI助手 | 医生（主任） |
|------|----------|-----------|
| 问病史 | AI通过语音/文字多轮采集，生成结构化病历 | 审核病历，补充关键问题 |
| 初步诊断 | 基于病历+知识库+历史病例给出建议 | 筛选、修改、做最终诊断 |
| 检查建议 | 推荐需要做的检查 | 确认/调整检查方案 |
| 治疗建议 | 推荐治疗方向（药物类别，不含剂量） | 确认/修改治疗方案 |
| 术后随访 | 回答患者常规问题，收集症状，提醒复查 | 处理AI上报的异常和紧急情况 |
| 全程管理 | 按医生制定的方案执行提醒和随访 | 制定管理方案，调整计划 |

### 渠道策略

患者通过微信小程序扫码进入（不接管医生私人微信）。
新患者出院/门诊后扫码注册，老患者在复查时引导迁移。

**这不是：**
- 诊所管理系统（ABC诊所管家的赛道 — 我们不做收费/药品/医保）
- 通用医疗聊天机器人（不了解患者具体情况的）
- 取代医生（AI只做初级工作，医生做所有决策）

## 三、真正的竞品

### 中国

| 竞品 | 定位 | 与我们的关系 |
|------|------|------------|
| **杏仁医生/未来医生AI工作室** | "随访老患者"、"智能小护士AI随访"、"康复随访智能体" | **最直接的竞品** — 明确做医生的术后随访AI |
| **好大夫在线** | 医患沟通平台，有AI辅助回复 | 平台拥有流量，不是医生拥有。医生是平台的"打工人" |
| **京东健康（京医智能体）** | 专家医生AI分身，为患者提供AI代答 | 平台级，医生没有控制权，数据属于京东 |
| **支付宝AI健康管家/名医AI分身** | 名医AI克隆，患者可以"问AI版的专家" | 同上——平台拥有，医生是内容来源不是用户 |
| **讯飞"智医随行"** | 医院级AI助手，含随访功能，覆盖神经外科 | 医院所有，不是医生个人工具 |
| **医生的现状（最大"竞品"）** | 深夜语音回复、选择性忽略、模板回复、找护士/助理帮忙分诊 | 我们要替代的就是这个 |

### 海外

| 竞品 | 定位 | 与我们的关系 |
|------|------|------------|
| **Hippocratic AI (Raven)** | 术后出院患者AI随访Agent，主动打电话跟进 | 方向最接近，但是电话而非微信，且面向美国市场 |
| **Artera Care Navigator** | 患者沟通自动化平台，多渠道（短信/电话/邮件） | 通用平台，不做专科深度 |
| **Commure Engage** | 数字化护理路径（digital care journeys），术后随访自动化 | 面向医院/健康系统，不面向个体医生 |

### 核心发现

**截至2026年3月20日，中国市场没有明显的主导产品在做"医生个人拥有的、微信原生的、专科AI助手"。**

- 平台型（好大夫/京东/支付宝）→ 医生不拥有数据和患者关系
- 医院型（讯飞/左手医生）→ 医院所有，医生换医院就没了
- 最近的竞品（杏仁医生/未来医生）→ 需要深入调研其实际功能和市场占有率

**我们的机会：医生拥有的、微信原生的、专科深度的AI助手。**

### 核心差异化原则：医生数据主权

所有平台型竞品（好大夫/京东/支付宝）的商业模式都建立在"平台拥有医生数据"之上。
医院型竞品（讯飞/左手医生）的数据属于医院，不属于医生。
我们做出根本性的不同承诺：

**1. 医生的数据完全归医生所有**
- 患者资料、病历记录、对话历史、随访数据 — 所有数据的所有权归医生个人
- 平台不使用医生数据训练通用模型、不向第三方共享、不用于商业分析
- 医生删除账号 = 数据彻底删除，不保留

**2. 医生的知识可以完全导入导出**
- 医生积累的专科随访Playbook、常见问答、回复模板、患者管理经验 — 随时可导出
- 支持标准格式导出（JSON/PDF/Markdown），可迁移到任何其他系统
- 从其他系统迁入同样无障碍 — 不设置迁移壁垒

**3. 所有数据属于个人，不属于平台**
- 这不只是隐私政策上的一句话，而是架构层面的设计原则
- 医生换医院、换平台、甚至停用我们的产品，数据完整带走
- 我们的商业模式建立在"工具价值"上（帮医生省时间），不建立在"数据锁定"上

**为什么竞品做不到这一点：**

| 竞品 | 数据归属 | 为什么不会改 |
|------|---------|------------|
| 好大夫/京东/支付宝 | 平台所有 | 数据是平台估值的核心资产，让医生带走=削弱自己 |
| 讯飞/左手医生 | 医院所有 | 卖给医院信息科的产品，医生不是客户 |
| 杏仁医生 | 平台所有（推测） | SaaS+平台模式，医生数据是产品壁垒 |
| **我们** | **医生所有** | **我们卖的是工具，不是数据。医生用得好自然续费，不需要锁定** |

这三条原则是我们的**品牌承诺**，也是与平台型竞品（好大夫/京东/支付宝）的
根本性差异化立场。他们不是"做不到"，而是"做了会损害自己的核心商业模式"。
对于新进入的SaaS竞品，这三条是可以从第一天就采用的——所以我们的优势在于
**先建立这个信任，并用产品力维持它**。

## 四、产品设计：AI助手的三段式工作流

AI助手像医生助手一样，覆盖**诊前问诊→诊中辅助→诊后随访**全流程。

### 三段式核心流程

```
诊前（第一优先）：
  患者扫码 → AI像医生助手一样语音问诊 → 生成结构化病历
  → 患者确认 → 传给医生
  "如果病人医生在看到病人前就有一份相对比较好的病例，他就会很舒服"

诊中（第二优先）：
  医生看到结构化病历 + AI初步诊断/鉴别/检查/治疗建议
  → 医生筛选确认 → 做最终决策
  "我在里面只要筛选，或者再加强一些，来决定下面的整治"

诊后（第三优先）：
  患者继续问诊 → AI基于病历+医嘱自动回复常规问题
  → 医生制定管理方案 → 提醒双方复诊/用药调整
  "到复诊、用药调整时会提醒双方，执行病人随访方案"
```

### 专科知识架构：大模型 + Skills + RAG + Context

医生说："某一个专科的疾病引起的症状有限的，可以做几百几千个问题的知识库"。
我们不需要建传统知识图谱，用三层架构让LLM拥有专科能力：

```
┌─────────────────────────────────────────────┐
│  Skills（专科提示词知识）                       │
│  神经科问诊话术、诊断规则、危险信号、治疗方案      │
│  → 告诉LLM"怎么问、怎么判断"                   │
│  → 医生可看/可改/可导入导出                     │
├─────────────────────────────────────────────┤
│  RAG（病例历史匹配）                           │
│  医生确认过的历史病例，embed后做相似匹配           │
│  → 新患者来了，找到相似的以前病例给LLM参考        │
│  → 用得越多，匹配越准（医生的"经验"在积累）       │
│  → 医生可看/可改/可导入导出                     │
├─────────────────────────────────────────────┤
│  Context（患者上下文）                         │
│  这个患者的病历、用药、恢复阶段、对话历史          │
│  → 每次回复都带上，AI"了解"这个患者              │
│  → 医生可看/可修正                             │
└─────────────────────────────────────────────┘
```

**为什么比传统知识图谱好：**

| | 知识图谱（左手医生） | 我们的 Skills + RAG + Context |
|---|---|---|
| 构建成本 | 巨大（6000+疾病实体关系） | 低（markdown提示词+病例积累） |
| 更新方式 | 需要专人维护图谱 | 医生确认病例自动积累 |
| 医生个性化 | 不能 | 天然实现（RAG匹配的是这个医生的病例） |
| 专科深度 | 广而浅 | 深而专（按亚专科写skills） |

**医生说"要有自己核心专科小模型在手才能安全"** — 不是训练模型权重，
而是积累专科知识库+病例库。这些才是真正属于医生的"小模型"。

### 医生可见可控的知识管理

**核心原则：AI知道的一切，医生都能看到、编辑、导入、导出。**

**1. Skills 管理 — 医生的专科知识库**

存储在 `doctor_knowledge_items` 表（已有），按类别分类：

| 类别 | 说明 | 示例 |
|------|------|------|
| `interview_guide` | 问诊指导 | "头痛该怎么问：先问发作方式，突发→问SAH..." |
| `diagnosis_rule` | 诊断规则 | "头痛+恶心+视乳头水肿→颅高压" |
| `red_flag` | 危险信号 | "雷击样头痛→蛛网膜下腔出血" |
| `treatment_protocol` | 治疗方案 | "脑膜瘤术后常规：激素减量方案..." |
| `custom` | 医生自定义 | 任何医生想让AI知道的知识 |

前端界面：

```
设置 → 我的知识库
├── 问诊指导 (12条)          [+ 添加]
│   ├── 头痛问诊话术          [查看] [编辑] [删除]
│   ├── 肢体无力问诊话术       [查看] [编辑] [删除]
│   └── ...
├── 诊断规则 (8条)           [+ 添加]
├── 危险信号 (5条)           [+ 添加]
├── 治疗方案 (6条)           [+ 添加]
│
├── [导入知识库] ← JSON/Markdown
├── [导出全部]  ← JSON/Markdown
└── 共 31 条知识 · 最后更新 3月20日
```

医生写的直接给AI用 — 修改问诊话术=改变AI怎么问，修改诊断规则=改变AI怎么判断。

**2. Case History 管理 — 医生的病例库**

```
设置 → 我的病例库
├── 总计 47 例 · [筛选: 脑血管▼ 已确认▼] [搜索...]
│
├── #001 头痛+恶心 → 脑膜瘤     [已确认] [查看] [编辑] · 被AI引用12次
├── #002 突发头痛 → SAH          [已确认] [查看] [编辑] · 被AI引用8次
├── #003 肢体无力 → 脊髓压迫     [初步]   [查看] [编辑] · 被AI引用3次
│
├── [导入病例]  ← JSON/CSV 批量导入
├── [导出全部]  ← JSON/CSV 带诊断+结局+备注
└── [从历史病历生成] ← 已确认的病历自动转为case_history
```

**3. Patient Context 可见性 — 医生看到AI用了什么**

查看患者时显示"AI上下文"面板：

```
患者：张三（动脉瘤夹闭术后第10天）
│
├── AI当前使用的上下文：
│   ├── 病历摘要        [展开] [修改]
│   ├── 出院医嘱        [展开] [修改]
│   ├── 当前用药        [展开] [修改]
│   ├── 匹配的相似病例：#002 SAH术后, #015 动脉瘤随访
│   └── 使用的知识条目：头痛问诊话术, 脑血管病鉴别
│
└── 医生随时可以修改AI"知道"什么
```

**4. 一键导出全部**

```
设置 → 我的数据
├── 导出知识库    → JSON + Markdown
├── 导出病例库    → JSON + CSV
├── 导出患者数据  → JSON（病历+对话+随访）
├── 导出全部      → ZIP 一键打包
│
└── "所有数据属于您。随时带走，无需理由。"
```

## 五、诊后随访：AI助手的第三段工作

诊后随访是AI助手的持续工作 — 就像住院医值班时替主任回答患者的常规问题。

不是通用医疗问答。是**基于这个患者的手术记录、出院医嘱、当前用药、恢复阶段**来回答。

```
患者问："刀口有点肿正常吗？"

通用AI回答：
  "术后伤口轻微肿胀是正常的，如果持续加重请就医。"
  （← 无用。不了解这个患者）

我们的AI回答：
  "张阿姨，您3月15日做的动脉瘤夹闭手术，现在是术后第5天，
   轻微肿胀是正常的恢复过程。如果出现以下情况请立即联系张医生：
   伤口周围明显发红/渗液/体温超过38°C。
   您的下次复查CT是3月25日，请记得按时复查。"
  （← 知道手术类型、日期、当前恢复阶段、下次复查安排）
```

**Day 1 价值：** 如果第一周不能把医生的夜间消息处理时间砍掉至少1小时，这不是产品。

### 每个患者的最小上下文

AI必须知道的：

| 字段 | 说明 | 必需 |
|------|------|------|
| 患者身份 | 姓名 + 授权联系人（家属代发消息） | 是 |
| 诊断/手术 | 类型 + 日期 + 侧别/部位 | 是 |
| 当前恢复阶段 | 术后第X天 / 恢复期 / 稳定期 | 是 |
| 出院医嘱 | 医生批准的出院指导（注意事项/活动限制/饮食） | **是（核心）** |
| 当前用药 | 药名 + 剂量 + 疗程 + 过敏史 | 是 |
| 正常 vs. 危险信号 | 当前阶段预期症状 vs. 需紧急上报的信号 | **是（安全基础）** |
| 下次复查/随访 | 时间 + 检查项目 | 是 |
| 最近医生决策 | 上次医生亲自回复的内容（保持一致性） | 推荐 |

**神经外科专科补充字段（按需）：**
- 手术亚型：动脉瘤/脑膜瘤/脊柱/外伤/分流术/癫痫
- 关键影像日期
- 当前神经功能状态
- 医生关注的危险信号（如：突发剧烈头痛/呕吐/意识改变 — 作为LLM上下文，由LLM判断是否上报）

### 消息分级机制

| 级别 | 触发条件 | 处理方式 | 示例 |
|------|---------|---------|------|
| **绿色** | 常规问题，出院医嘱已覆盖 | AI自动回复 | "药什么时候吃"、"可以洗头吗" |
| **黄色** | 需要专业判断但不紧急 | AI起草回复 → 等医生确认发送 | "伤口有点痒"、"想提前复查" |
| **红色** | LLM判断为紧急/危险情况 | 立即通知医生 + 给患者急救指导 | "突然剧烈头痛"、"呕吐"、"抽搐"、"肢体无力" |

### 工作流

```
患者微信发消息
  ↓
系统识别患者身份 → 加载患者上下文（手术/用药/恢复阶段）
  ↓
消息分级（绿/黄/红）
  ├── 绿色：AI基于出院医嘱自动回复（标注"AI助手"）
  ├── 黄色：AI起草回复 → 推送给医生 → 医生确认/修改 → 发送
  └── 红色：立即推送给医生 + 给患者发急救提示
  ↓
医生端展示：
  - 今日绿色 32条（已处理）
  - 今日黄色 5条（待确认）
  - 今日红色 1条（紧急）
  ↓
所有对话记录存档 + 审计日志
```

## 六、风险与安全

### 安全架构：LLM驱动一切，医生提供上下文和最终决策

架构很简单 — **所有判断和回复都由 LLM 完成**，不需要硬编码规则引擎：

```
患者发消息
  ↓
LLM（带患者完整上下文：病历+医嘱+用药+恢复阶段）
  ↓
生成回复 + 判断紧急程度（绿/黄/红）
  ├── 绿色：AI直接回复（标注"AI助手"）
  ├── 黄色：AI起草回复 → 推送给医生 → 医生确认/修改 → 发送
  └── 红色：通知医生 + 给患者急救提示
```

**医生的三个角色：**

| 角色 | 说明 |
|------|------|
| **提供上下文** | 录入或导入病历、出院医嘱、用药、复查安排。LLM基于这些数据回答患者 |
| **设置偏好** | 哪些类型的问题AI可以直接回复、哪些需要确认、什么算紧急 |
| **最终决策** | 黄色消息一键确认/修改后发送；红色消息亲自处理 |

**为什么不需要额外的规则引擎：**
- LLM本身能理解语义和紧急程度（"头有点晕" vs "突然剧烈头痛站不住"）
- LLM能结合患者上下文判断（术后第3天的轻微头痛 ≠ 术后第30天的新发头痛）
- 医生通过 prompt/偏好设置告诉LLM什么该自动回复、什么该上报
- 如果LLM判断不确定，默认升级（宁可多问医生，不可漏报）

**关于"绿色消息AI直接发送"的说明：**
这不代表"没有医生参与"。医生参与体现在：
- 医生提供了上下文数据（病历/医嘱）→ LLM基于这些数据回答
- 医生设置了偏好（哪些类型可以自动回复）→ LLM遵循这些边界
- 医生随时可以查看AI已发送的回复 → 发现问题可以纠正
正确的表述是：**医生定义边界和处理例外，LLM在边界内自主运行。**

### 致命风险

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **错误安慰** | AI把恶化的症状判断为"正常恢复" | LLM通过 prompt 被指导宁可过度上报。不确定时默认升级给医生 |
| **错误上下文** | 搞错患者/搞错术后天数/用了过期的医嘱 | 每次回复前验证患者身份+上下文时效 |
| **患者以为是医生回复** | 信任/期望风险 + 法律风险 | **必须明确标注"AI助手回复"，不得伪装成医生** |
| **合规** | 互联网诊疗监管（2022）：AI不能替代医生做诊断/处方。AI生成内容标识办法（2025.9）：必须标注AI生成 | AI只回答出院医嘱范围内的教育/随访问题，不做新诊断/处方 |
| **数据安全** | 患者健康数据通过微信传输 | 端到端加密存储，审计日志，最小权限 |

### LLM的 prompt 边界（通过 system prompt 约束）

### 医生侧 vs. 患者侧的能力边界

| | 对医生（AI助手角色） | 对患者（AI助手角色） |
|---|---|---|
| 诊断 | **可以**给初步诊断+鉴别建议（医生筛选确认） | **不可以**直接告诉患者新诊断结论 |
| 检查 | **可以**推荐检查方案 | **可以**提醒已安排的检查时间 |
| 治疗 | **可以**建议治疗方向（药物类别） | **不可以**修改用药或推荐新药 |
| 病历 | **可以**生成结构化病历 | **可以**展示患者自己的病历摘要 |
| 随访 | **可以**汇报患者状态+上报异常 | **可以**回答基于医嘱的常规问题 |
| 急救 | **可以**标记红色紧急 | **可以**引导拨打120/联系医生 |

**简单记忆：对医生是"建议"，对患者是"解释已有医嘱"。医生做决策，患者收到的是决策的结果。**

**容错原则：** LLM不确定时默认升级为黄色（推给医生）。
宁可多打扰医生一次，不可漏掉一个危险信号。
医生可以根据使用经验调整LLM的"自信度阈值"。

## 七、分阶段交付（定位不变，交付分期）

定位从第一天就是"AI助手"，但交付分三期。每期独立可用，叠加后形成完整能力。

### 第一期（1-2个月）：AI问诊 + 结构化病历

医生的第一优先需求："看到病人前就有一份好病历"

| 模块 | 具体内容 |
|------|---------|
| **患者扫码入口** | 微信小程序扫码注册（姓名+性别+年龄+电话） |
| **AI语音问诊** | 像医生助手一样多轮采集：主诉→现病史→既往史→个人史→婚育史→家族史 |
| **结构化病历生成** | 门诊/住院病历格式，患者确认后传给医生 |
| **医生审核工作台** | 查看病历、补充追问、确认/修改 |
| **专科知识库（初版）** | 神经科问诊话术 + 常见症状追问清单（Skills） |

### 第二期（2-3个月）：AI诊断建议

医生说："希望有初步诊断、鉴别诊断以及进一步治疗的建议，我只要筛选"

| 模块 | 具体内容 |
|------|---------|
| **AI诊断建议** | 基于结构化病历 + 知识库 + 历史病例，给出初步诊断+鉴别+检查+治疗方向 |
| **医生筛选确认** | 对每个建议：确认/修改/拒绝，一键操作 |
| **病例库积累** | 每个确认的诊断自动进入case_history，RAG匹配越来越准 |
| **外部模型桥接** | 接口预留对接成熟的诊断API（医生提到"有成熟公司做这块"） |

### 第三期（3-4个月）：诊后随访 + 全程管理

| 模块 | 具体内容 |
|------|---------|
| **诊后自动随访** | AI基于病历+医嘱回答患者常规问题（绿/黄/红分级） |
| **全程管理方案** | 医生制定管理计划，患者可见可执行 |
| **双向提醒** | 复诊、用药调整到期时提醒医患双方 |
| **患者端"我的健康"** | 患者看到自己的病历时间线+待办+下次复查 |

### 不做的

| 不做 | 原因 |
|------|------|
| 处方/具体用药剂量 | 合规红线（无过敏/用药数据时不安全） |
| 收费/医保/进销存 | ABC诊所管家的赛道，不是我们的 |
| 全科覆盖 | 先做深神经科，再扩展 |

选择同样痛点的专科（高焦虑恢复期 + 重复性随访问题 + 可规则化的危险信号 + 强私域患者池）：

## 十、渠道决策：不能用私人微信bot

Codex 特别强调的关键决策：

**不能在医生的私人微信上做自动回复bot。** 原因：
- 微信封号风险
- 无法控制消息格式和标注
- 无审计日志
- 无法标注"AI生成"（违反2025.9法规）
- 患者无法区分是医生还是AI

**正确方案：** 出院时给患者扫码进入受控通道（WeCom客服账号/小程序/服务号）。
这个通道：
- 医生品牌化（"张医生的AI助手"）
- 所有AI回复标注"AI助手"
- 完整审计日志
- 医生可以随时接管对话
- 患者明确知道是AI+医生协作

#### 差异2：专科深度 vs. 通用广度

```
其他产品的AI：
  患者："手术后头痛正常吗？"
  AI："术后头痛很常见，建议观察，严重时就医。" ← 通用回答，什么手术都适用

我们的AI：
  患者："手术后头痛正常吗？"
  AI："王叔叔，您3月10日做的右侧大脑中动脉动脉瘤夹闭术，
       现在是术后第10天。这个阶段的轻微头痛是正常的，
       多和颅骨愈合有关。但如果出现突发剧烈头痛、呕吐、
       意识模糊，请立即拨打120并联系张医生。
       您的下次CTA复查是3月25日。" ← 知道手术类型、日期、恢复阶段、复查安排
```

**为什么竞品做不到：** 通用平台要覆盖几十个专科，无法为每个亚专科
（动脉瘤/脑膜瘤/脊柱/外伤/分流术/癫痫）定制恢复时间线和危险信号规则。
我们从一个专科做深，每个手术类型都有精确的Playbook。

#### 差异3：医生品牌 vs. 平台品牌

```
京东健康：患者看到的是"京东健康·张医生智能体" → 患者认的是京东
好大夫：  患者看到的是"好大夫在线·张医生" → 患者认的是好大夫
讯飞：    患者看到的是"XX医院智能随访" → 患者认的是医院
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
我们：    患者看到的是"张医生的AI助手" → 患者认的是张医生本人
```

**为什么这很重要：** 对于有私域患者池的专科医生，品牌就是信任。
患者信的是"张主任"，不是某个平台。我们帮医生强化个人品牌，
而不是把医生变成平台的流量入口。

#### 差异4：场景聚焦 — 诊后随访，不是全流程

| 产品 | 试图覆盖 | 结果 |
|------|---------|------|
| ABC | 诊前+诊中+诊后+管理+营销+供应链 | 每个都做，每个都不深 |
| 京东/好大夫 | 挂号+问诊+处方+随访+科普+电商 | 平台逻辑，医生是工具人 |
| 讯飞 | 预防+治疗+康复+随访+科研 | 要医院采购，部署周期长 |
| **我们** | **AI助手全流程：问诊→病历→建议→随访** | **一个助手角色覆盖全流程，医生只做决策** |

医生不需要又一个"什么都能做"的平台。他需要的是：
**每天晚上200条微信消息变成3条需要他亲自看的。**

### 优势分类：护城河 vs. 先发优势

**差异化立场（平台型竞品不愿复制，新竞品可以但我们先建立信任）：**

| 立场 | 对平台型竞品（好大夫/京东） | 对新SaaS竞品 |
|------|------------------------|------------|
| **数据归医生** | 他们的估值建在数据上，不愿放弃 | 可以做，但我们已经先建立了信任 |
| **知识可导出** | 靠锁定维持粘性，不愿开放 | 可以做，我们先做 |
| **不依赖数据锁定** | 收入模式决定了不会改 | 可以做，但需要找到替代商业模式 |

**先发优势（竞品可以做但我们先做深）：**

| 优势 | 说明 |
|------|------|
| 医生品牌化 | 患者看到的是"张医生"不是平台。竞品理论上也能做，但会削弱平台品牌 |
| 专科术后深度 | 通用平台覆盖广但不深。我们先在神经外科做透，但竞品如果决定也做是可以的 |
| 独立于医院/平台 | 先发优势，但不是绝对壁垒 |
| 轻量部署 | 竞品理论上也能做轻量版 |

**诚实的说：** 专科深度、绿黄红分级、患者级上下文回复——这些都是产品优势，
不是护城河。我们的差异化立场是**数据归医生、知识可导出、商业模式不靠锁定**。
这三条对平台型竞品来说意味着重写商业模式（他们不愿意），
对新竞品来说可以采用（但我们先建立了信任和口碑）。
