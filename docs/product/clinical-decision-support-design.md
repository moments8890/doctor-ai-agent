## Product Value Proposition

A doctor-centric AI clinical assistant where:
- **The patient is the data source, not the user.** AI interviews patients to
  collect complete clinical information — patients just answer questions.
- **The AI learns from THIS doctor.** Diagnostic suggestions reference the
  doctor's own case history and diagnostic patterns, not just generic textbooks.
- **The doctor makes the decisions.** AI presents structured records with
  preliminary diagnoses, workup, and treatment — doctor confirms, modifies, or
  overrides with one tap.
- **WeChat-native delivery.** No app install — patient scans QR and goes. This
  is the wedge in China's private practice market.

## Architecture Decisions (from Eng Review)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Architecture path** | Path B: LLM-first + structured overlay | Ship fast, validate with real doctors. Invest in structured domain model (Path A) later if PMF validated. |
| **Build sequence** | Dashboard first (P3 → P1 → P2) | Doctor gets value on day 1. Dashboard gives a surface to display diagnosis on. |
| **Knowledge base** | Simplified: extend existing `skills/` + `case_history` table | No abstract interfaces, no question trees as code. LLM does reasoning. |
| **Diagnosis module** | Single `src/domain/diagnosis.py` (~200 lines) | Split when it grows past 300 lines. No premature abstraction. |
| **Diagnosis execution** | Async background task via APScheduler | Record saves immediately, diagnosis runs async. Graceful degradation if LLM fails. |
| **Case matching** | Lightweight RAG: in-memory embedding search | Embed on write, cosine similarity at query time. No vector DB needed at <1,000 cases. |
| **Treatment suggestions** | Drug classes only, no specific doses | Clinical safety: no allergy/medication/renal data collected yet. |
| **F2.4 External APIs** | Deferred entirely | PHI risk: free-text clinical history is identifying. Needs legal framework. |
| **Growth loop** | `confidence_status` field on case history | Entries tracked as preliminary → confirmed_with_evidence → revised. Weighted matching. |
| **Persistence** | New `diagnosis_results` table | Stores AI suggestions + doctor decisions separately for audit trail + growth loop. |
| **LLM model** | Qwen3:32b (dev/prod), Qwen3.5:9b (local) | Single model family for consistency. |
| **DRY fix** | Move `FIELD_LABELS` to `completeness.py` | Duplicated in `interview_turn.py` and `interview_summary.py`. |

## Constraints

- **Production product** — must be reliable, secure, and clinically credible
  for daily use by neurosurgeons
- **Audience:** Private practice / clinic neurosurgeons in China
- **Delivery:** WeChat mini-program (patient) + Web UI (doctor)
- **Clinical safety:** AI suggestions are advisory only — doctor has final
  authority on all diagnoses and treatment decisions; audit trail required
- **Data privacy:** Patient data isolation before doctor accepts; PHI handling
  per Chinese healthcare data regulations (《健康医疗大数据安全管理办法》)
- **Build on existing codebase** — Phase 0 is stable, extend don't rewrite
- **Python 3.9** compatible, `from __future__ import annotations`
- **LLM:** Qwen3:32b (dev/prod), Qwen3.5:9b (local) via Ollama at LAN server
- **No Alembic migrations** — `create_tables()` until first production launch
- **Neurosurgery first** — architecture must support multi-specialty expansion
  but initial content is neuro-only

## Build Sequence (Revised per Eng Review)

### Phase P3: Doctor Dashboard & Review Workflow ✅ COMPLETE
**Roadmap:** Enables doctor-side of F1.1 handoff + F2.1 review

**Incoming records view:**
- REST endpoint `GET /api/doctor/incoming` — patient records with status
- Statuses: 等待 (waiting) → 问诊中 (in-interview) → 待审核 (ready-for-review) →
  诊断中 (diagnosis running) → 诊断完成 (diagnosis ready) → 已审核 (reviewed)
- Patient queue sidebar with status badges, sorted by urgency (red flags first)
- Click to open full record + diagnosis

**Review workflow:**
- Split view: structured medical record (left) + AI diagnosis/workup/treatment (right)
- For each differential diagnosis: confirm ✓ / modify ✏️ / reject ✗ buttons
- For workup items: approve / modify / remove
- For treatment suggestions: approve / modify / replace / remove
- Add own diagnosis / workup / treatment
- Red flag alert banner when urgent findings detected
- "Finalize" button — saves confirmed version, updates case history (growth loop)
- Every action logged to audit trail

**New DB table: `diagnosis_results`**
```
diagnosis_results:
  id, record_id (FK medical_records), doctor_id,
  ai_differentials (JSON),       # what AI suggested
  ai_workup (JSON),              # AI-suggested tests
  ai_treatment (JSON),           # AI-suggested treatment (drug classes only)
  confirmed_differentials (JSON), # what doctor approved
  confirmed_workup (JSON),
  confirmed_treatment (JSON),
  red_flags (JSON),
  status (pending/confirmed/rejected),
  created_at, confirmed_at
```

**Files:**
- `src/channels/web/doctor_dashboard.py` — new dashboard endpoints
- `frontend/web/` — dashboard UI components (queue, review panel, timeline)
- `src/db/models/diagnosis.py` — DiagnosisResult ORM model
- `src/db/crud/diagnosis.py` — CRUD operations
- API endpoints for status updates, diagnosis confirmation, audit logging

### Phase P1: Knowledge Base + Case History ✅ COMPLETE
**Roadmap:** F4.2 — provides context for diagnosis pipeline
**Dependency:** None (can start immediately, but P3 ships first)

**Simplified approach (Path B / LLM-first):**
- Extend existing `skills/neurology/` with new `diagnosis.md` skill file —
  neurology-specific diagnosis prompts, red flag rules, common differentials
  (same pattern as `clinical_signals.md` and `structuring.md`)
- Add `case_history` table with `confidence_status` field
- Case matching via lightweight RAG: embed on write, in-memory cosine similarity

**New DB table: `case_history`**
```
case_history:
  id, doctor_id, patient_id (nullable FK),
  chief_complaint, final_diagnosis, key_symptoms (JSON array),
  treatment, outcome, notes,
  confidence_status (preliminary/confirmed_with_evidence/revised/superseded),
  revised_by_id (nullable FK self-reference),
  embedding (JSON — vector for RAG matching),
  created_at
```

**Seed data:** 20-30 realistic neurosurgery cases covering:
- Headache variants (tension, migraine, intracranial mass/meningioma)
- Stroke/TIA presentations
- Spinal cord compression / disc herniation
- Peripheral neuropathy / carpal tunnel
- Epilepsy / seizure disorders
- Trigeminal neuralgia, hydrocephalus

**Files:**
- `src/domain/knowledge/skills/neurology/diagnosis.md` — diagnosis prompt
- `src/db/models/case_history.py` — case history ORM model
- `src/db/crud/case_history.py` — CRUD + embedding search

### Phase P2: Clinical Decision Support Pipeline ✅ COMPLETE
**Roadmap:** F2.1 + F2.2 + F2.3 — core value proposition
**Dependency:** P1 (case history for matching), P3 (UI to display results)

**Diagnosis pipeline (single module):**
- Trigger: interview confirmed → record saves → background task queued
- Process:
  1. Load record structured fields
  2. Query case_history via RAG (embed chief_complaint + present_illness, cosine match)
  3. Build prompt: record + matched cases + skills/neurology/diagnosis.md
  4. Call LLM (Qwen3:32b) → structured `DiagnosisResult`
  5. Save to `diagnosis_results` table
  6. Update record status → diagnosis_ready
- Graceful degradation: if LLM fails/times out, record available for manual review

**Output: `DiagnosisResult`**
```
differentials: List[Differential]  # condition, confidence, reasoning
workup: List[WorkupItem]           # test/imaging, rationale, urgency level
treatment: List[TreatmentItem]     # drug CLASS (not dose), intervention, duration
red_flags: List[str]               # urgent findings requiring immediate action
case_references: List[CaseRef]     # matched historical cases with outcomes
disclaimer: str                    # always: "AI建议仅供参考，最终诊断由医生决定"
```

**Safety guardrails:**
- Never auto-confirm any diagnosis — doctor MUST explicitly confirm
- Red flag detection triggers prominent UI alerts
- Drug classes only (e.g., "Corticosteroid for cerebral edema") — no specific doses
- Confidence scores: <50% = "consider", 50-80% = "likely", >80% = "highly suggestive"
- Audit trail: every AI suggestion + doctor decision logged with timestamp
- Fallback: if LLM fails, show structured record without diagnosis

**Files:**
- `src/domain/diagnosis.py` — single module: pipeline + models + safety (~200 lines)
- `src/domain/knowledge/skills/neurology/diagnosis.md` — diagnosis prompt

### Phase P4: Patient Onboarding & Multi-Modal Input ⚠️ PARTIAL
**Roadmap:** F1.2 (QR entry) + F1.3 (voice, photo, file)
**Dependency:** Can run in parallel with P1/P2

**Patient entry (F1.2 / ADR 0017):**
- QR code generation per doctor — doctor shares QR, patient scans to enter
- Patient self-registration: name, phone, gender, age (minimal friction)
- WeChat OpenID binding: link WeChat identity to patient record
- Access model: patient belongs to the doctor who generated the QR

**Voice input (F1.3):**
- STT integration for patient interview — voice-primary with text fallback
- Provider options: WeChat native STT, Tencent Cloud ASR, browser Speech API
- Latency target: <2s from speech end to text appearance
- Error recovery: fallback to text input if STT fails

**Photo/file input (F1.3):**
- Extend existing `vision_import.py` to patient context
- Patient uploads: prior medical records, lab results, imaging reports
- OCR/vision processing → extracted text added to interview context

**Files:**
- `src/channels/wechat/qr.py` — QR generation and scanning handler
- `src/domain/patient_onboarding.py` — self-registration flow
- `src/domain/voice/` — STT integration module
- Extend `src/agent/tools/` with photo/file upload tools for patient agent

### Phase P5: Doctor-Patient Communication Loop ⚠️ PARTIAL
**Roadmap:** F3.1 (post-visit portal) + F3.2 (treatment plan) + F3.3 (reminders)
**Dependency:** P3 (doctor must finalize review before patient sees results)

**Post-visit patient portal (F3.1):**
- Patient views: confirmed diagnosis, treatment plan, upcoming appointments
- Follow-up questions: patient asks → AI triage → doctor (if needed)
- Upload new results: lab/imaging results auto-attached to patient record

**Treatment plan visibility (F3.2):**
- Doctor creates structured treatment plan during review (P3)
- Patient sees actionable checklist: medications, lifestyle, appointments, warning signs

**Bilateral reminders (F3.3):**
- Extend existing task/notification system to patient side
- Delivery: WeChat template messages (both doctor and patient)

**Files:**
- `src/channels/web/patient_portal.py` — extend existing portal
- `src/domain/treatment_plan.py` — structured treatment plan model
- `src/domain/triage.py` — AI message triage (patient→AI→doctor routing)
- Extend `src/infra/notify/` for patient-side notifications

### Phase P6: Outcome Tracking & Growth Loop (ADR 0021)
**Roadmap:** F4.1 — longitudinal patient tracking
**Dependency:** P5 (requires treatment plan + communication loop)

**Treatment plan execution tracking:**
- Patient self-reported: medication compliance, symptom changes, side effects
- Doctor-entered: clinical outcomes, follow-up exam findings, imaging results

**Growth loop — the system gets smarter:**
- Every confirmed diagnosis + outcome → case history entry (confidence_status=preliminary)
- When outcome confirmed → update to confirmed_with_evidence (weighted 3x in matching)
- If diagnosis revised → mark original as revised, link to corrected entry
- This is the moat: longer use = better suggestions = harder to leave

**Files:**
- `db/models/outcome.py` — outcome tracking ORM
- `src/domain/outcome_tracking.py` — longitudinal data management
- Extend case history with outcome data

## Data Flow Diagram

```
Patient submits interview
        │
        ▼
Record saves immediately (status: ready_for_review)
        │
        ├──────────────────────────────────┐
        ▼                                  ▼
Doctor refreshes dashboard          Background task: diagnosis
(sees record without diagnosis)     1. RAG: embed → match case history
        │                           2. LLM: record + cases + prompt → result
        │                           3. Save to diagnosis_results
        │                           4. Update status → diagnosis_ready
        │                                  │
        ▼                                  ▼
Doctor refreshes again ──────────→ Sees record + AI diagnosis
        │
        ▼
Doctor confirms/modifies/rejects
        │
        ├─ Save confirmed version to diagnosis_results
        ├─ Create case_history entry (preliminary)
        ├─ Log to audit trail
        └─ Update status → confirmed
```
