患者端聊天工作流 — 现状 + Tier B 计划

2026-04-26 · 基于 Codex 90% 一致后的设计 · /Volumes/ORICO/Code/doctor-ai-agent/docs/patient-chat-tier-b.html

患者发一条消息，系统决定：(1) 这是不是急症 (2) 应该走哪条处理路径 (3) 给医生生成什么样的草稿。患者本身几乎不会收到 AI 自动回复，所有内容都进医生的审核队列。这就是“桥梁不是服务”的产品定位。

1. 患者端聊天里 LLM 在哪里被调用

每条患者消息进入 POST /chat 后，LLM 被调用的地方今天有三处：

LLM 调用 确定性逻辑（无 LLM） 入口/出口 已知 BUG（Codex 发现）
POST /chat · src/channels/web/patient_portal/chat.py:122

① 入口：保存患者消息 + 鉴权

读取 PATIENT_CHAT_INTAKE_ENABLED 开关。开关 ON 走新版 intake 路径，OFF 走 legacy fire-and-forget。

↓
signal_flag.detect() → triage.classify() · LLM 调用 #1

② 信号标记检测 LLM

用 intent/triage-classify.md 判断这条消息是不是急症。返回 5 类之一。如果是 urgent → 立刻 short-circuit，给患者发去“您的医生将尽快查看并回复您，请稍候”（Tier A 已修），同时给医生生成 priority=critical 草稿。

↓ 非急症继续
triage.classify() · LLM 调用 #2 BUG

③ 第二次分类 LLM

这就是 Codex 发现的 bug：同一条消息被 classify() 又跑了一次。signal_flag.py:12 的注释写“我们不双重调用 LLM”，但 chat.py:282 和 chat.py:342 加起来确实调用了两次。每条消息多花一次 LLM token。

↓
chat_state.evaluate_entry() · 确定性

④ 状态机决策 无 LLM

三种状态：idle（什么都没发生）/ intake（正在填问诊）/ qa_window（医生回复后 30 分钟答疑窗口）。是否进入 intake 取决于：分类是否 symptom_report + 置信度阈值（≥0.65）或词典命中（≥0.50）。

↓ 如果进入 intake
_extract_and_append_fields() · chat.py:437

⑤ 字段抽取 无 LLM

第二个 bug：当前实现把整条消息无条件塞进 chief_complaint。所以“我老婆腹泻”会被存成患者本人的主诉。这是 Tier B 要修的核心问题之一。

↓
dedup._llm_chief_complaint_similarity() · LLM 调用 #3（条件性）

⑥ 去重相似度（仅在阈值满足时）LLM

主诉 + 现病史都填了 → 检查 24 小时内有没有相似主诉的旧记录。一条小 LLM 计算文本相似度，决定 auto_merge / patient_prompt / 新建。

↓
draft_reply.generate_draft_reply() · LLM 调用 #4（异步后台）

⑦ 医生草稿生成 LLM

最重的一次 LLM。用 intent/followup_reply.md + 完整的 L0-L7 层栈生成给医生的回复草稿。这条草稿患者看不到，只有医生在审核队列里能看到。

↓
MessageDraft 入库 + 优先级标注

⑧ 出口：草稿落库

style_guard 检测 defer-to-doctor 用语 → priority 解析（urgent/critical）→ 医生在审核 tab 看到带优先级标的草稿。

2. 6 层 Prompt 系统是什么

项目里所有 LLM 调用都用一个统一的“层组合”系统。每个 intent（followup_reply / patient-intake / diagnosis 等）声明它需要哪些层，prompt_composer.py 按顺序拼起来。

L0 · Style Guard

prompts/common/style_guard.md

反 AI 味、禁词列表（“立即去急诊”、“根据知识库”等）、defer-to-doctor 模式。所有患者/医生可见的文字都要经过它。

L1 · Identity

domain/system base prompt

“你是某医生的 AI 助手”之类的身份定义。所有 flow 都用。

L2 · Specialty

domain config (神经外科 / 内科 etc.)

专科上下文。不同专科有不同的判断尺度。

L3 · Task

prompts/intent/<intent>.md

具体任务说明。比如 followup_reply.md、patient-intake.md、triage-classify.md。

L4 · Doctor Rules (KB)

doctor 自己写的 knowledge_items

这位医生的个人规则库。“开浦兰副作用 1-2 周”、“术后 VAS≤6 正常”之类。每位医生不一样。

L4p · Persona

doctor_personas 表

医生的语言风格画像。让 AI 写出来“像这位医生”，不是“像 AI”。

L5 · Examples

data/style/example_pool.json (564 条)

从 MedDG/meddialog 真实医患对话提取的范例，按主诉聚类。给当前消息选 2-3 条最相近的范例喂进去。

L6 · Patient Context

patient_lifecycle/triage_context.py

这位患者的稳定史 + 最近一次记录（诊断、过敏史、随访计划）。用来让回复有患者背景，不是“通用回复”。

L7 · Input

运行时拼装

当前这条消息本身。在 conversation_mode 下还包括对话历史。

3. Codex 发现的核心问题

把 Tier A 之前的 bugs 加上 Codex 这一轮发现，共 4 个必须修：

①

双重 LLM 分类（性能 + 一致性）

signal_flag.detect 调一次 classify，主分发又调一次。每条消息双倍 LLM token，且两次结果可能不一致（temperature 0.1 不是 0）。

修法：合并成一次调用，返回更丰富的结果对象 TriageAnalysis。

②

chief_complaint 被无条件污染

chat.py:443 把第一条消息直接塞进 chief_complaint。不管这条消息是患者本人的还是“我老婆/我宝宝”的，全都当成本人主诉存进医疗记录。

修法：抽取前先做 subject 检测，proxy 情况下不写 chief_complaint，让医生核实。

③

MessageDraft 没地方放标注

想给医生 UI 加“用药决策 / 化验解读 / 代述 / 多问题”的桶标，目前 schema 里没有任何字段能存。要么硬塞进 seed_source 字符串里（hack），要么不持久化（不可靠）。

修法：迁移加 4 个列。

④

Prompt 契约冲突

triage-classify.md 第 8 行明确说“化验单解读”归 informational，但我们想把它独立成 interpretation 桶。代码改了 prompt 不改，模型还是按老规矩分类。

修法：先改 prompt，再改 enum/schema。

4. 设计前后对比：分类 vs 标注

Codex 最关键的建议：不要把新的语义往 TriageCategory 里塞。它今天既是“分类标签”又是“路由开关”。把这两件事拆开。

❌ 我原来的方案

class TriageCategory(Enum):
    informational
    symptom_report
    side_effect
    general_question
    urgent
    med_decision      # ← 新增
    interpretation    # ← 新增

问题：
• 每个 TriageCategory 消费者都要处理新枚举
• 置信度<0.7 会被强制重写成 general_question，新枚举消失
• 状态机只对 symptom_report 进入 intake，新枚举走 legacy

✅ Codex 建议的方案

class TriageAnalysis:
    route_category: TriageCategory   # 5 路由类不变
    confidence: float
    review_bucket: str               # ← 新：标注层
        # none | med_decision | interpretation
    signal_tags: List[str]           # ← 新：副信号
        # distress | proxy | multi_question
    needs_disambiguation: bool       # ← 新：是否需要追问

优点：
• 路由稳定，5 类不变
• 标注独立，不被低置信度覆盖
• 一次 LLM 调用产出全部信号
• 多个 signal 可以共存（胸痛 + 自伤倾向 = urgent + distress）

5. 实施顺序（90% 与 Codex 一致）

共 5 步，约 1.5-2 天工作量。前 2 步是基础设施（必须先做），后 3 步是 Tier B 安全增强。B4（追问澄清）和 B5（多问题检测）从原计划砍出，单独排期—— B4 是首个“非信号标记”的 AI 主动回复，是产品策略而非工程优化，单独想清楚。

P1~半天

前置 合并双重 LLM 分类

为什么先做：没有 TriageAnalysis 这个新结构，B1/B2/B3 都没地方挂标注。同时省一半 LLM 成本。

改动：
  • triage.py · 新 dataclass TriageAnalysis，classify() 返回它
  • signal_flag.py · 改成纯 helper is_signal_flag(analysis: TriageAnalysis) -> bool，不再调 LLM
  • chat.py:282 · 删掉第一次 signal_flag_detect 调用，只在 chat.py:342 调一次 classify
  • 更新 signal_flag.py:12 注释，让它和实现一致
P2~30 分钟

前置 Alembic 迁移：MessageDraft 加 4 列

为什么先做：没有列就没法持久化标注，UI 桶就是“空中楼阁”。

# 加到 message_drafts 表
review_bucket String(32) # none|med_decision|interpretation
patient_subject String(16) # self|proxy
signal_tags JSON # ["distress","multi_question"]
multi_question Boolean # 冗余存一份方便 UI 索引
B1
+B6~半天

Tier B 安全 Triage prompt 重写 + intake 词典扩展

为什么：用药决策（“能不能停药”）和化验解读（“这个值什么意思”）今天被 AI 自由发挥，是临床风险点。术后/孕期/伤口照片关键词在 intake 词典里没有，导致这些消息走不进结构化问诊。

改动：
  • triage-classify.md · 重写为输出 TriageAnalysis JSON，明确 review_bucket 定义；删掉“化验单归 informational”那行
  • chat_state.py:25-27 · 词典扩展加术后/伤口/孕周/照片标记词
  • draft_reply.py · 把 review_bucket 写到 MessageDraft
B2~1-2 小时

Tier B 安全 心理痛苦确定性扫描

为什么：语料里“打针三天没好…现在想死的感觉都有”这种消息，分类器会抓“打针没好”丢掉“想死”。心理信号要让医生第一眼就看到。

改动：
  • 新模块 distress_scan.py · 关键词列表 {想死, 受不了, 活不下去, 绝望, 不想活, 没意思}
  • chat.py 在 classify 后调用 → 把 distress 加进 signal_tags
  • priority.py · 任何 signal 含 distress → priority=critical（不覆盖原有 route，胸痛 + 想死 仍然 route=urgent）
B3~1-2 小时

Tier B 安全 代述检测 + 抽取保护

为什么：第 ② 个 bug 的修法。“我宝宝拉肚子”不应该被记录成“患者本人主诉拉肚子”。

改动：
  • 新模块 subject_detector.py · 关键词扫 {宝宝, 孩子, 我妈, 我爸, 我老公, 我老婆, 家人, 我儿子, 我女儿}
  • _extract_and_append_fields() · 检测到 proxy → 不写 chief_complaint，只写 present_illness（保留原文）
  • _intake_threshold_met() · 调整为 proxy 情况下 present_illness 单字段够阈值（避免永远卡住）
  • 医生 UI 提示“代述，请核实主诉对象”

6. 推迟到下次会话的项目

B4 · 追问澄清回合

“我不舒服” → AI 回复“您指的是哪里不舒服？大概多久了？”

为什么推迟：这是除信号标记外第一次 AI 主动给患者发文字。是产品策略变更，要单独想清楚循环防护、与 doctor 通知的关系等。需要在 ChatSessionState 加持久化标志。

B5 · 多问题检测

一条消息有 3 个以上 “吗/?” → 标 multi_question

为什么推迟：UI 工效项不是安全项。“吗”在中文里太常见，简单计数会误报，需要更精细的句子切分逻辑。

Tier C / D （之后）

  • Tier C：图片输入支持。语料里术后伤口照、化验单照、孕检 B 超照高频出现，今天空文字消息直接 422 报错。
  • Tier D：孕期 / 术后状态作为患者持久标志贯穿所有 triage 调用；“上次您说”的医生连续性记忆。
