Guiding Principle

The 19 issues from the review aren’t equal. For a small-scale MVP (1–5 specialist doctors, private practice, real patient data), think in three layers:

  1. Will it lose data?
  2. Will it expose data?
  3. Will doctors trust it?

Everything else can wait.

Week 1 — The Two Issues That Determine If This System Is Safe

If you only fix two things this week, make it these.

Priority 1: AI Record Confirmation Gate

Files affected: routers/records.py, services/structuring.py, db/models.py, db/crud.py

Why this is the most important change:

Right now the LLM writes directly to medical_records with no human review step. A hallucinated dosage or negated diagnosis goes straight into the permanent record. The feasibility report states “所有内容在医生确认后形成结构化数据” — but the code does not enforce this.

The fix — two-step flow:

Doctor dictates
    ↓
LLM structures → stored in `pending_records` table (NOT medical_records)
    ↓
System sends preview back to doctor via WeChat:

    "已生成病历草稿，请确认：
     主诉：胸闷气促3天
     诊断：心力衰竭 NYHA III级
     处理：调整利尿剂剂量
     回复【确认】保存，或直接修改内容发送"

    ↓
Doctor replies "确认"
    → record moves from pending_records → medical_records
Doctor replies with edits
    → re-structure → show preview again
No reply in 10 minutes
    → flag as abandoned, notify doctor

New table needed:

CREATE TABLE pending_records (
    id          VARCHAR(64) PRIMARY KEY,
    doctor_id   VARCHAR(64) NOT NULL,
    patient_id  INT,
    draft_json  TEXT NOT NULL,          -- structured MedicalRecord JSON
    raw_input   TEXT,                   -- original dictation for reference
    status      VARCHAR(16) NOT NULL DEFAULT 'awaiting_confirmation',
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    expires_at  DATETIME NOT NULL,      -- created_at + 10 minutes
    INDEX (doctor_id, status)
);

Time: 3–4 days. Non-negotiable for clinical use.

Priority 2: Durable Message Handling

Files affected: routers/wechat.py, db/models.py, db/crud.py

Why:

If the process crashes between WeChat receiving a message and the LLM finishing, the message is silently lost. The doctor thinks it was saved. It wasn’t.

New table needed:

CREATE TABLE pending_messages (
    id          VARCHAR(64) PRIMARY KEY,
    doctor_id   VARCHAR(64) NOT NULL,
    raw_content TEXT NOT NULL,
    msg_type    VARCHAR(16) NOT NULL DEFAULT 'text',  -- text, voice, image
    status      VARCHAR(16) NOT NULL DEFAULT 'pending',
    error       TEXT,
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    processed_at DATETIME,
    INDEX (status, created_at)
);

The fix — in routers/wechat.py, before any asyncio.create_task():

msg_id = await db.create_pending_message(
    doctor_id=doctor_id,
    raw_content=raw_text,
    status="pending"
)
asyncio.create_task(process_and_mark(msg_id, ...))

The background task:

async def process_and_mark(msg_id: str, ...):
    try:
        await process_message(...)
        await db.update_pending_message(msg_id, status="done")
    except Exception as e:
        await db.update_pending_message(msg_id, status="failed", error=str(e))
        log.error(f"Message {msg_id} processing failed: {e}")

On startup — recovery job:

async def recover_pending_messages():
    """Pick up any messages that were pending before last restart."""
    cutoff = datetime.utcnow() - timedelta(seconds=60)
    pending = await db.list_pending_messages(status="pending", older_than=cutoff)
    for msg in pending:
        asyncio.create_task(process_and_mark(msg.id, ...))

No message is ever silently dropped, even across restarts.

Time: 2–3 days.

Week 2 — Make It Production-Deployable

Switch to MySQL with Real Persistence

Do this as one unit — the database switch and backup setup belong together.

Step 1: Provision Tencent Cloud TencentDB for MySQL (not a Docker container).

Step 2: Set DATABASE_URL in the production .env file:

DATABASE_URL=mysql+aiomysql://user:pass@host:3306/doctor_ai

Step 3: Mount MySQL data on a separate persistent disk (if self-hosting).

Step 4: Set up a daily backup cron to Tencent COS:

# /etc/cron.d/doctor-ai-backup
0 2 * * * appuser mysqldump -u root -p$DB_PASS doctor_ai \
  | gzip | coscmd upload - backups/$(date +\%Y-\%m-\%d).sql.gz

Time: Half a day once TencentDB is provisioned.

Real Production Deployment

Replace dev.sh vm-up with systemd + nginx.

/etc/systemd/system/doctor-ai.service:

[Unit]
Description=Doctor AI Agent API
After=network.target mysql.service

[Service]
User=appuser
WorkingDirectory=/app
ExecStart=/app/.venv/bin/uvicorn main:app \
    --host 127.0.0.1 \
    --port 8000 \
    --workers 2
Restart=always
RestartSec=5
EnvironmentFile=/app/.env

[Install]
WantedBy=multi-user.target

/etc/nginx/sites-available/doctor-ai:

server {
    listen 443 ssl;
    server_name your-domain.com;

    ssl_certificate     /etc/ssl/doctor-ai.crt;
    ssl_certificate_key /etc/ssl/doctor-ai.key;

    location / {
        proxy_pass         http://127.0.0.1:8000;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$host$request_uri;
}

No --reload. No Cloudflare tunnel. Auto-starts on reboot.

Time: 1 day.

Add the Audit Log Table

New table:

CREATE TABLE audit_log (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    ts            DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    doctor_id     VARCHAR(64) NOT NULL,
    action        VARCHAR(32) NOT NULL,   -- READ, WRITE, DELETE, LOGIN
    resource_type VARCHAR(32),            -- patient, record, task
    resource_id   VARCHAR(64),
    ip            VARCHAR(45),
    trace_id      VARCHAR(64),
    ok            BOOLEAN NOT NULL DEFAULT TRUE,
    INDEX (doctor_id, ts),
    INDEX (resource_type, resource_id)
);

New helper in services/audit.py:

async def audit(
    doctor_id: str,
    action: str,
    resource_type: str | None = None,
    resource_id: str | None = None,
    ok: bool = True,
) -> None:
    async with get_session() as session:
        session.add(AuditLog(
            doctor_id=doctor_id,
            action=action,
            resource_type=resource_type,
            resource_id=resource_id,
            ok=ok,
        ))
        await session.commit()

Call from every router on every patient record create, read, update, delete.

Time: 1 day.

Week 3 — Close the Remaining Safety Gaps

By now the system is reliable enough for real use. Week 3 closes the remaining gaps.

Separate Patient vs. Doctor WeChat Pipelines

File: routers/wechat.py

Route messages from patient WeChat OpenIDs to a sandboxed handler that has no access to delete_patient, list_patients, or any tool that touches other patients’ data. Only the doctor-origin pipeline gets full agent access.

Add a lookup at the top of the WeChat message handler:

sender_role = await db.get_sender_role(open_id)  # "doctor" | "patient" | "unknown"

if sender_role == "doctor":
    await handle_doctor_message(msg, doctor_id)
elif sender_role == "patient":
    await handle_patient_message(msg, patient_id)  # sandboxed, no destructive tools
else:
    await reply(msg, "您好，请联系您的医生开通服务。")

Time: 1 day of routing logic, assuming doctor/patient OpenID mapping exists in the DB.

Flush Session Turns to DB Synchronously

File: services/session.py

After the pending_message fix, ensure conversation turns are written to the DB before returning the response — not lazily via _flush_pending_turns().

# Before returning the response to the caller:
await db.append_conversation_turn(doctor_id, turn)  # awaited, not fire-and-forget

Makes process restarts invisible to the doctor. Their context is always recoverable from the DB.

Time: Half a day.

Start the NMPA Regulatory Conversation

This is not engineering work, but it has a 12–24 month lead time.

The system performs clinical decision support (risk scores, guideline alerts), which likely classifies it as a Class II medical device under China’s 医疗器械监督管理条例. Operating without NMPA registration exposes the operator to legal liability if any patient harm occurs.

Actions this week: 1. Engage a Chinese regulatory affairs consultant to formally classify the product 2. Scope the MVP to stay within the lowest-risk classification — Phase 1 (transcription and documentation only) is lower risk than Phase 2/3 (risk stratification, decision support) 3. Ensure all AI outputs visible to the doctor are labeled “仅供参考，请医生确认” (for reference only, please confirm)

Time: No engineering time. Business/legal action only.

What to Explicitly Defer

These are real issues but not MVP blockers at small scale (1–5 doctors).

Issue Why Defer
Encryption at rest Enable TencentDB encrypted storage (one checkbox at provisioning) — covers you at infrastructure level without app changes
APScheduler race condition Only matters with multiple server instances; don’t run multiple instances at MVP
Multi-tenancy isolation audit Do as a careful PR code review pass, not a full rewrite
Context compression (120-char limit) Real issue but no acute harm at small patient volumes; fix before scaling
Regex fallback alerting Add logging now, skip alerting infrastructure until you have more users
Knowledge crawl validation Don’t ship Phase 3 features until Phase 1 + 2 are stable
Confidence scores on AI output Important UX improvement but not a safety blocker once the confirmation gate exists
Nice-to-have features All of them — none until this list is done
