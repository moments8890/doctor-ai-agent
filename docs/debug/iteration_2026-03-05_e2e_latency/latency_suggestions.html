Latency Improvement Suggestions

Based on: metrics_summary.json, report.md, tencent_lkeap_vs_baselines_2026-03-06.md

Core Finding

LLM inference is 99.33% of all latency. DB (0.18%), persist (0.20%), and dispatch overhead (0.14%) are noise. Do not optimize those layers until the LLM path is addressed.

Current Baselines (per chat turn)

Backend Avg/turn Notes
Local Ollama (single-req) ~1.15s Best case, no concurrency
Tencent LKEAP (cloud) ~2.34s Best cloud option
DeepSeek public API ~3.82s
Full E2E run (1003 cases) 6.29s Ollama under concurrent load

The full-run avg of 6.29s is ~5.5x higher than the single-request Ollama baseline of 1.15s. This gap is almost entirely queueing under concurrent load — not model size or prompt complexity.

Recommended Actions (in order)

1. Fix concurrency degradation first

The existing plan lists this as Priority 3, but the 5.5x load-induced degradation makes it the most urgent bottleneck in practice.

  • Benchmark E2E replay at workers = 1, 2, 4, 6, 8
  • Select operating point by weighted score: 0.6*p95 + 0.2*p99 + 0.2*wall_time
  • Hypothesis: 2–3 workers will give the best p95 tradeoff; beyond that, queueing dominates

2. Expand deterministic fast-paths

Only 5 of 3,073 dispatches triggered structure_medical_record — meaning 99.8% of turns are conversational, not structuring. Yet they all pay the same ~6.3s LLM cost. Many are simple commands (list, delete, schedule, query) that don’t need a model call.

Target: 30%+ of turns served in <100ms with zero LLM calls. Expected impact: ~20–30% reduction in avg turn latency.

Expand fast-paths for: - list patients / list tasks - delete patient (already partially done) - query records by name - schedule appointment / complete task

Trigger only on high-confidence command patterns; fall back to LLM on ambiguity.

3. Model tiering for remaining LLM turns

After fast-paths reduce total LLM call count, split remaining turns: - Simple conversational intents (ack, clarify, confirm) → lighter/faster model - Structuring and complex agent dispatch → current heavy model

Tencent LKEAP at 2.34s/turn is the best cloud option if local hosting is replaced. Add config knobs for per-intent model selection.

4. Token budget control

Cap max_tokens on non-clinical responses (command acks, confirmations, list outputs). Apply only to non-structuring turns to avoid truncating medical data.

Expected impact: ~10% reduction in avg LLM call duration.

What NOT to Do

  • Do not optimize DB or persistence — they are 0.18% of latency
  • Do not add LLM response caching — medical notes are unique per turn
  • Do not switch models before fixing the concurrency problem — you will benchmark the wrong bottleneck

Suggested Iteration Sequence

Iteration A: Concurrency benchmarking + fast-path expansion - Run E2E at multiple worker counts; pick optimal setting - Expand fast-paths for list / delete / query / schedule / task-complete - Re-run full 1003-case corpus; compare avg/p95/p99 before and after

Iteration B: Model tiering - Route low-complexity turns to faster model - Verify no regression in E2E pass rate

Iteration C: Token budget + observability - Cap response length for non-clinical turns - Add run-scoped metrics endpoint (per optimization_plan.md Priority 5)
