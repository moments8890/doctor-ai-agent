Optimization Execution Plan (Ranked by Expected Latency Impact)

Goal

Reduce /api/records/chat end-to-end latency (especially P95/P99) for human-language E2E workloads.

Baseline (from this run)

  • Dataset: 1003 cases, 3079 chat turns
  • Turn latency: avg 6287ms, p95 8265ms, p99 11409ms, max 22319ms
  • Attribution: LLM inference dominates (99.33% of chat trace time)

Priority 1 — Split model tiers (fast router vs heavy structuring)

  • Why: Dominant cost is agent.chat_completion; routing simple intents with a lighter model should cut most turns.
  • Actions:
    1. Keep current heavy model for complex structuring only.
    2. Route simple deterministic intents (list/query/delete/task commands) to lighter local model.
    3. Add config knobs for per-intent model selection.
  • Success criteria:
    • P95 turn latency drops >= 25%
    • No regression in task/patient CRUD E2E pass rate
  • Risk:
    • Intent misclassification causing wrong path.
  • Guardrail:
    • Fallback to heavy model when confidence is low.

Priority 2 — Add deterministic fast-paths to bypass LLM

  • Why: Many admin/task/patient operations are syntax-driven and currently still pay LLM cost.
  • Actions:
    1. Expand existing fast-path handlers for: delete/list/query/schedule/complete commands.
    2. Normalize Chinese ordinal parsing and same-name disambiguation without model roundtrip.
    3. Keep LLM only for ambiguous/narrative inputs.
  • Success criteria:

    • = 30% of turns served by non-LLM fast path in corpus replay

    • Fast-path turn avg < 100ms (server-side)
  • Risk:
    • Rule conflicts with free-text clinical notes.
  • Guardrail:
    • Trigger fast-path only on high-confidence command patterns.

Priority 3 — Concurrency tuning for model serving

  • Why: Outliers align with concurrent pressure; p99 spikes during high parallel load.
  • Actions:
    1. Benchmark replay under workers=2/4/6/8.
    2. Measure wall time + p95/p99 simultaneously.
    3. Set recommended default worker/concurrency in scripts.
  • Success criteria:
    • Keep throughput near max while reducing p99 >= 15%
  • Risk:
    • Lower worker count improves tail latency but hurts total runtime.
  • Guardrail:
    • Select operating point by weighted score: 0.6*p95 + 0.2*p99 + 0.2*wall_time.

Priority 4 — Prompt/response token budget control

  • Why: Long prompts and verbose outputs increase inference latency.
  • Actions:
    1. Minimize runtime prompt extensions for non-structuring flows.
    2. Cap generation length for command/ack responses.
    3. Add concise response mode for E2E replay.
  • Success criteria:
    • Avg turn latency -10% without semantic regressions
  • Risk:
    • Over-truncation may omit critical medical details.
  • Guardrail:
    • Apply cap only to non-clinical command responses.

Priority 5 — Observability improvements for run-scoped diagnosis

  • Why: In-memory /api/admin/observability window is bounded; run-level JSONL analysis is external.
  • Actions:
    1. Add run_id filter endpoint reading persisted JSONL.
    2. Add per-run breakdown dashboard (LLM/DB/router split).
    3. Export p50/p95/p99 by case family (BASE/AUTO/SAMENAME).
  • Success criteria:
    • One-command report generation for each E2E run
  • Risk:
    • Higher metrics I/O if unbounded.
  • Guardrail:
    • Time-window and run_id filters with max limits.

Recommended Execution Sequence (2-3 iterations)

  1. Iteration A:
    • Implement Priority 2 fast-path expansion + tests.
    • Replay half/full and capture before/after.
  2. Iteration B:
    • Implement Priority 1 model-tier split + fallback.
    • Tune with Priority 3 benchmark matrix.
  3. Iteration C:
    • Apply Priority 4 token controls.
    • Add Priority 5 run-scoped observability endpoint.

Verification Protocol

  • Functional:
    • ./dev.sh test integration-full
    • ./dev.sh e2e full
  • Performance:
    • Compare baseline vs new run on:
      • Turn avg/p95/p99
      • Max latency
      • Total replay wall time
    • Require no drop in pass rate.

Deliverables

  • Updated metrics JSON + markdown report per run
  • Slow-trace CSV snapshots before/after
  • Change log with latency deltas and rollback switch
