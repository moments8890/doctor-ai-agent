Doctor AI Agent — Performance Analysis & Fix Plan

Context: 3s production latency using Tencent LKEAP (DeepSeek-v3) model Date: 2026-03-06

Summary

The 3s latency has two primary causes that stack on each other:

  1. ROUTING_MAX_TOKENS=220 is too low — tool call output gets truncated → structured_fields is None → a second sequential LLM call is fired for structuring
  2. A new AsyncOpenAI HTTP client is created per request — no TCP/TLS connection reuse

Everything else is additive overhead on top of these two root causes.

Root Cause #1 — ROUTING_MAX_TOKENS=220 Forces a Second LLM Call

Files: services/agent.py:233, routers/records.py:560-578

What happens

# services/agent.py
routing_max_tokens = int(os.environ.get("ROUTING_MAX_TOKENS", "220"))

When a doctor dictates clinical content, the add_medical_record tool call must return a JSON object with up to 9 fields. A typical dictation like:

“张三，男，58岁，胸闷气促3天，BNP 980，EF 50%，心衰III级，给予利尿剂调整”

produces tool call output like:

{
  "name": "add_medical_record",
  "arguments": {
    "patient_name": "张三",
    "chief_complaint": "胸闷气促3天",
    "history_of_present_illness": "BNP 980pg/mL，EF 50%，心衰III级",
    "diagnosis": "心力衰竭 NYHA III级",
    "treatment_plan": "利尿剂调整"
  }
}

That JSON alone is 200–400 tokens. Plus the required chat reply (“好的，张三的情况已记录。”) adds ~20 tokens. 220 tokens is frequently not enough, causing the output to be truncated. Truncated JSON → json.loads() fails → args = {} → structured_fields = None.

Then in routers/records.py:

# routers/records.py:560-578
if intent_result.structured_fields:
    # Fast path: use tool call output directly — ONE LLM call total ✅
    record = MedicalRecord(**{k: fields.get(k) for k in MedicalRecord.model_fields})
else:
    # FALLBACK: fires a SECOND LLM call 🔴
    record = await structure_medical_record("\n".join(doctor_ctx))

Two sequential LLM calls to LKEAP ≈ 1.5s + 1.5s = 3s. This matches exactly the observed production latency.

Fix

ROUTING_MAX_TOKENS=600

600 tokens comfortably covers any realistic clinical dictation in the tool call output plus a short chat reply. This single env var change likely cuts latency from 3s to ~1.5s.

Root Cause #2 — New AsyncOpenAI Client Created Per Request

Files: services/agent.py:218-223, services/structuring.py (same pattern)

What happens

# services/agent.py — inside dispatch(), called on EVERY request
client = AsyncOpenAI(
    base_url=provider["base_url"],
    api_key=os.environ.get(provider["api_key_env"], "nokeyneeded"),
    timeout=float(os.environ.get("AGENT_LLM_TIMEOUT", "45")),
    max_retries=0,
)

AsyncOpenAI wraps an httpx.AsyncClient. Creating it fresh on every request means:

  • New TCP connection + TLS handshake to LKEAP on every single request (~100–300ms)
  • No HTTP/2 multiplexing or connection reuse across requests
  • The old client and its connection pool are abandoned without await client.aclose()

The same pattern exists in services/structuring.py for the structuring call.

Fix — module-level singletons

# services/agent.py — at module level, created ONCE at import time
_CLIENT_CACHE: dict[str, AsyncOpenAI] = {}

def _get_client(provider_name: str, provider: dict) -> AsyncOpenAI:
    if provider_name not in _CLIENT_CACHE:
        _CLIENT_CACHE[provider_name] = AsyncOpenAI(
            base_url=provider["base_url"],
            api_key=os.environ.get(provider["api_key_env"], "nokeyneeded"),
            timeout=float(os.environ.get("AGENT_LLM_TIMEOUT", "45")),
            max_retries=0,
        )
    return _CLIENT_CACHE[provider_name]

# Inside dispatch(), replace:
#   client = AsyncOpenAI(...)
# With:
#   client = _get_client(provider_name, provider)

Apply the same change in services/structuring.py. One persistent HTTP connection pool per provider eliminates the per-request TLS handshake. Saves ~150–300ms per request.

Root Cause #3 — Full Tool Schema Sends ~2,500 Extra Input Tokens

File: services/agent.py — _selected_tools(), default AGENT_TOOL_SCHEMA_MODE=full

What happens

The _TOOLS list has detailed Chinese descriptions per parameter. All 8 tools × 5–10 fields each with full descriptions ≈ 2,000–3,000 input tokens just for the tool schema.

def _selected_tools() -> List[dict]:
    mode = os.environ.get("AGENT_TOOL_SCHEMA_MODE", "full")  # ← defaults to "full"
    if mode in {"compact", "lite"}:
        return _TOOLS_COMPACT  # strips all "description" keys (~400 tokens)
    return _TOOLS              # full descriptions (~2,500 tokens)

The compact version already exists and strips all description fields. For intent routing (which only needs tool names and parameter names), this is fully sufficient.

Fix

AGENT_TOOL_SCHEMA_MODE=compact
AGENT_ROUTING_PROMPT_MODE=compact

Drops input tokens from ~2,500 to ~400 for the routing call. Saves ~100–200ms per request.

Root Cause #4 — Synchronous File I/O Blocks the Event Loop on Every Request

File: services/observability.py — _append_jsonl()

What happens

Every trace_block() exit calls add_span() which calls _append_jsonl() — a synchronous blocking file write:

def _append_jsonl(path: Path, payload: Dict[str, Any]) -> None:
    path.parent.mkdir(parents=True, exist_ok=True)
    with path.open("a", encoding="utf-8") as f:          # ← blocking I/O
        f.write(json.dumps(payload, ensure_ascii=False) + "\n")

A single /api/records/chat request passes through these trace_block() calls:

  • records.chat.agent_dispatch
  • llm.agent.chat_completion
  • records.chat.structured_fields_to_record
  • records.chat.persist_record
  • structuring.load_prompt
  • structuring.chat_completion
  • structuring.parse_response

That is 5–7 synchronous file writes per request, each blocking the asyncio event loop. On a Tencent Cloud CVM with network-attached storage, each write can take 5–20ms — adding 35–140ms of cumulative blocking I/O per request, and stalling all other concurrent requests while the event loop is blocked.

Fix — async write queue

# services/observability.py
import asyncio
from asyncio import Queue

_SPAN_WRITE_QUEUE: Queue = Queue(maxsize=5000)

async def _disk_writer():
    """Background coroutine: drains the queue and writes to disk non-blocking."""
    while True:
        path, payload = await _SPAN_WRITE_QUEUE.get()
        try:
            path.parent.mkdir(parents=True, exist_ok=True)
            with path.open("a", encoding="utf-8") as f:
                f.write(json.dumps(payload, ensure_ascii=False) + "\n")
        except Exception:
            pass
        _SPAN_WRITE_QUEUE.task_done()

def _enqueue_jsonl(path: Path, payload: dict) -> None:
    """Non-blocking enqueue. Drops silently if queue is full."""
    try:
        _SPAN_WRITE_QUEUE.put_nowait((path, payload))
    except asyncio.QueueFull:
        pass  # drop span rather than block the event loop

# Replace _append_jsonl(...) calls in add_trace() and add_span() with _enqueue_jsonl(...)
# Start the writer in main.py lifespan:
#   asyncio.create_task(_disk_writer())

Saves ~50–140ms per request and stops event loop stalls.

Root Cause #5 — Knowledge Context Loaded Sequentially Before LLM Dispatch

File: routers/records.py:478-491

What happens

# Sequential: DB query completes FIRST, then LLM call starts
knowledge_context = ""
async with AsyncSessionLocal() as db:
    knowledge_context = await load_knowledge_context_for_prompt(db, doctor_id, body.text)

# LLM dispatch only starts AFTER knowledge is loaded
intent_result = await agent_dispatch(body.text, knowledge_context=knowledge_context)

If knowledge lookup takes 50–150ms (DB query + ranking), this is dead sequential time before the LLM call even begins.

Fix — in-memory cache with 5-minute TTL

Most doctors’ knowledge bases don’t change during a clinic session. Cache the result:

# services/doctor_knowledge.py
import time
from typing import Optional, Tuple

_KNOWLEDGE_CACHE: dict[str, Tuple[float, str]] = {}
_KNOWLEDGE_CACHE_TTL = 300  # 5 minutes

async def load_knowledge_context_for_prompt_cached(
    doctor_id: str, query: str
) -> str:
    now = time.time()
    cached = _KNOWLEDGE_CACHE.get(doctor_id)
    if cached and now - cached[0] < _KNOWLEDGE_CACHE_TTL:
        return cached[1]
    async with AsyncSessionLocal() as db:
        result = await load_knowledge_context_for_prompt(db, doctor_id, query)
    _KNOWLEDGE_CACHE[doctor_id] = (now, result)
    return result

Drops knowledge lookup from 50–150ms to ~0ms on cache hit. Saves ~100ms per request.

Root Cause #6 — maybe_auto_learn_knowledge Blocks the Response

File: routers/records.py:604-609

What happens

async with AsyncSessionLocal() as db:
    await save_record(db, doctor_id, record, patient_id)
    await maybe_auto_learn_knowledge(      # ← awaited before response is returned
        db, doctor_id, body.text,
        structured_fields=record.model_dump(exclude_none=True),
    )
# Doctor waits for this before receiving their confirmation
return ChatResponse(reply=reply, record=record)

maybe_auto_learn_knowledge may make additional LLM calls to extract knowledge. The doctor waits for this before their confirmation message is returned.

Fix — fire and forget

async with AsyncSessionLocal() as db:
    await save_record(db, doctor_id, record, patient_id)

# Return response immediately; knowledge learning runs in background
asyncio.create_task(
    _background_auto_learn(doctor_id, body.text, record.model_dump(exclude_none=True))
)
return ChatResponse(reply=reply, record=record)

async def _background_auto_learn(doctor_id: str, text: str, fields: dict) -> None:
    try:
        async with AsyncSessionLocal() as db:
            await maybe_auto_learn_knowledge(db, doctor_id, text, structured_fields=fields)
    except Exception as e:
        log(f"[Chat] background auto-learn failed doctor={doctor_id}: {e}")

Saves ~50–200ms per request (more if LKEAP is called for extraction).

Root Cause #7 — 6 History Messages Add ~3,000 Extra Input Tokens

File: routers/records.py:93

What happens

_ROUTING_HISTORY_MAX_MESSAGES = max(0, int(os.environ.get("ROUTING_HISTORY_MAX_MESSAGES", "6")))

Up to 6 prior messages are sent to the routing LLM. Each clinical message is 200–500 tokens. Six messages = up to 3,000 extra input tokens per routing call — all to resolve a pronoun (“他”) or a patient name from the prior turn, for which 2 messages suffice.

Fix

ROUTING_HISTORY_MAX_MESSAGES=2

Saves ~100ms per request (fewer input tokens = faster LKEAP response).

Root Cause #8 — LKEAP Connection Not Warmed Up at Startup

File: main.py — _warmup() function

What happens

async def _warmup(config: AppConfig):
    if config.routing_llm == "ollama" or config.structuring_llm == "ollama":
        # Ollama is warmed up...
    # ← No warmup for tencent_lkeap

Ollama is warmed up at startup. LKEAP is not. The first request after every deployment pays the full TCP connection + TLS handshake cost (~150–300ms extra on first request).

With the singleton client fix (Root Cause #2), subsequent requests reuse the connection — but only after the first request establishes it.

Fix — add LKEAP to the warmup

# main.py — inside _warmup(), after existing Ollama block:
if config.routing_llm == "tencent_lkeap" or config.structuring_llm == "tencent_lkeap":
    if config.tencent_lkeap_api_key:
        try:
            from services.agent import _get_client, _PROVIDERS
            client = _get_client("tencent_lkeap", _PROVIDERS["tencent_lkeap"])
            await client.chat.completions.create(
                model=config.tencent_lkeap_model,
                messages=[{"role": "user", "content": "hi"}],
                max_tokens=1,
            )
            log.info("[Warmup] LKEAP connection established")
        except Exception as e:
            log.warning(f"[Warmup] LKEAP warmup failed (non-fatal): {e}")

Eliminates cold-start penalty (~150–300ms on first request after restart).

Code Changes (Week 1)

1. Singleton AsyncOpenAI clients

In services/agent.py and services/structuring.py, replace per-request client creation with module-level singletons (see Root Cause #2 fix above).

Impact: ~200ms saved per request.

2. Async span writer

In services/observability.py, replace _append_jsonl() with a non-blocking queue and a background writer coroutine (see Root Cause #4 fix above). Start the coroutine in main.py lifespan.

Impact: ~100ms saved per request, eliminates event loop stalls under concurrent load.

3. Background maybe_auto_learn_knowledge

In routers/records.py, move maybe_auto_learn_knowledge to asyncio.create_task() after returning the response (see Root Cause #6 fix above).

Impact: ~100ms saved per request.

4. Knowledge context cache

In services/doctor_knowledge.py, add a 5-minute in-memory TTL cache for load_knowledge_context_for_prompt (see Root Cause #5 fix above).

Impact: ~100ms saved per request on cache hit.

5. LKEAP warmup

In main.py _warmup(), add a LKEAP warmup call after the Ollama block (see Root Cause #8 fix above).

Impact: Eliminates cold-start penalty on first request after deployment.

The remaining ~1,000ms is the irreducible cost of one LKEAP API round-trip. To go below 1s, the only options are: - Switch to a faster/smaller model for routing (e.g., DeepSeek-V2.5 or a flash variant) and use DeepSeek-V3 only for the structuring fallback - Enable streaming responses so the doctor sees output token-by-token instead of waiting for the full response
