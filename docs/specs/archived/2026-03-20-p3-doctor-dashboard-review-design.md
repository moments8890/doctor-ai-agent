# Design: P3 — Doctor Dashboard & Review Workflow ✅ DONE

> Generated by brainstorming session on 2026-03-20
> Status: APPROVED

## Goal

Add a review workflow so doctors can review, edit, and confirm structured
medical records from patient interviews. Reviews live inside the existing
Tasks tab as a new segment, mobile-first.

## Context

Phase P3 of the clinical decision support roadmap. The patient interview
pipeline (ADR 0016 / F1.1) is built — patients complete AI-guided interviews
that produce structured records. Doctors currently have no way to review
those incoming records, edit fields, or mark them as confirmed.

P3 ships before the diagnosis pipeline (P2). Diagnosis-related statuses and
the split-view AI suggestions panel are deferred until P2 lands.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Status model | 3 statuses for P3: 等待 → 待审核 → 已审核 | Diagnosis statuses (诊断中, 诊断完成) added when P2 lands |
| Architecture | New `review_queue` table (approach B) | Clean separation from medical_records; no coupling to pending_records scheduler |
| UI placement | Under Tasks tab, 3 segments | Avoids 5th nav item; reviews are "things to do" alongside tasks |
| Segment order | 待办 (default) \| 待审核 \| 已完成 | 待办 is doctor's home base; 待审核 is a filtered view |
| Reviews in 待办 | Yes — mixed with tasks | Doctor sees everything they need to do in one place |
| Review detail | Drill-down: structured fields → chat history → action bar | Mobile-first, same pattern as TaskDetailView |
| Actions | Confirm + inline edit only | Reject deferred until P5 communication loop |
| Editing | Tap field → inline edit | Matches 90% case: doctor fixes one field, not full form |
| Status transitions | Auto: interview complete → pending_review. Manual: doctor confirms → reviewed | No ambiguity; doctor explicitly owns the confirm action |
| Queue updates | Pull-to-refresh | Sufficient for private practice volume; real-time deferred |
| Reviewed retention | Query filter: 2 days in 已完成 tab; rows persist indefinitely | Doctor can revisit recent decisions; record always on patient timeline |
| Audit | Existing medical_record_versions (+ new old_structured column) + audit() | Minimal new infrastructure |

## Database Schema

### New table: `review_queue`

```sql
CREATE TABLE review_queue (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    record_id       INTEGER NOT NULL UNIQUE,
    doctor_id       VARCHAR NOT NULL,
    patient_id      INTEGER,
    status          VARCHAR NOT NULL DEFAULT 'pending_review',
    created_at      DATETIME NOT NULL,
    reviewed_at     DATETIME,
    FOREIGN KEY (record_id) REFERENCES medical_records(id),
    FOREIGN KEY (doctor_id) REFERENCES doctors(doctor_id),
    FOREIGN KEY (patient_id) REFERENCES patients(id)
);

CREATE INDEX ix_review_queue_doctor_status_created
    ON review_queue (doctor_id, status, created_at);
```

- `record_id` is UNIQUE — a record can only be in the queue once
- `patient_id` denormalized for efficient list queries (avoids 3-table join;
  matches `DoctorTask` pattern which also has its own `patient_id`)
- `status` values: `pending_review`, `reviewed`

### Existing column: `medical_records.needs_review`

This boolean column already exists but becomes redundant once `review_queue`
is the source of truth. **Migration rule:** `needs_review` is no longer read
or written by P3 code. It remains in the schema for backward compatibility
but is effectively dead. Do not sync it with `review_queue.status`.

### Schema addition: `medical_record_versions.old_structured`

The existing `MedicalRecordVersion` model only snapshots `old_content`,
`old_tags`, and `old_record_type`. Inline field edits modify the `structured`
JSON column, so we need to capture the old structured data for audit:

```python
# Add to MedicalRecordVersion model:
old_structured: Mapped[Optional[str]] = mapped_column(Text, nullable=True)  # JSON dict
```

This column is written when a structured field is edited via the review
PATCH endpoint. Existing version-creation code paths are unchanged.

### Lifecycle

1. Patient interview completes → summary saved to `medical_records` (existing)
2. After save → insert `review_queue` row with `status=pending_review`
3. Doctor reviews + confirms → `status=reviewed`, `reviewed_at=now()`
4. Query: `WHERE doctor_id=? AND status=pending_review ORDER BY created_at DESC`
5. 已完成 query: `WHERE status=reviewed AND reviewed_at > now() - 2 days`

## API Endpoints

Four new endpoints in `src/channels/web/ui/review_handlers.py`:

### 1. List review queue

```
GET /api/manage/review-queue?doctor_id=...&status=pending_review&limit=50
```

Returns records with patient name, chief complaint (truncated), created_at,
status. Default status: `pending_review`. Pass `status=reviewed` for 已完成
(filtered to last 2 days server-side).

### 2. Get review detail

```
GET /api/manage/review-queue/{queue_id}?doctor_id=...
```

Returns: structured record fields + interview chat history. Chat history
comes from `interview_sessions.conversation` (JSON array column), queried
by `patient_id` and cross-referenced with the record's `created_at`.
Single call gives the doctor everything for the review screen.

### 3. Confirm review

```
POST /api/manage/review-queue/{queue_id}/confirm?doctor_id=...
```

Sets `status=reviewed`, `reviewed_at=now()`. Logs to audit trail via
`audit("review.confirmed", ...)`. Returns updated item.

### 4. Update record field (inline edit)

```
PATCH /api/manage/review-queue/{queue_id}/record?doctor_id=...
Body: { "field": "chief_complaint", "value": "头痛3天" }
```

Updates the specific structured field on the underlying `medical_records`
row. Valid `field` values are the 14 keys from `OUTPATIENT_FIELD_META` in
`domain/records/schema.py`: `department`, `chief_complaint`,
`present_illness`, `past_history`, `allergy_history`, `personal_history`,
`marital_reproductive`, `family_history`, `physical_exam`, `specialist_exam`,
`auxiliary_exam`, `diagnosis`, `treatment_plan`, `orders_followup`.
Unknown field names return 422. Creates a version entry in
`medical_record_versions` (with `old_structured` snapshot) for audit.
Calls `audit("review.field_edited", ...)`. Returns updated record.

### Pattern notes

- All endpoints use `_resolve_ui_doctor_id()` for auth
- All endpoints scope queries by `doctor_id` (data isolation)
- Rate limiting via `enforce_doctor_rate_limit(doctor_id, scope="review")`

## Frontend Design (Mobile-First)

### UI placement

Reviews live inside `TasksSection.jsx` as a new segment. No new nav item —
bottom nav stays at 4 (AI助手, 患者, 任务, 设置).

### Segment control

Three segments in the pill-toggle (matches existing Tasks pattern):

```
待办 (default) | 待审核 | 已完成
```

- **待办**: pending tasks + pending reviews, mixed, sorted by time
- **待审核**: pending reviews only (filtered view)
- **已完成**: completed tasks + reviewed records (last 2 days)

### Visual differentiation: task vs review items

| Aspect | Task Item | Review Item |
|--------|-----------|-------------|
| Avatar shape | Square (4px radius) — task type icon | Circle — patient name initial |
| Title | Task title (e.g. "陈建国 随访") | Patient name + "问诊记录" |
| Subtitle | Task content / patient name | Chief complaint (truncated) |
| Badge | Date only | Orange "待审核" chip + date |
| Drill-down | → TaskDetailView (existing) | → ReviewDetail (new) |

### Review detail screen (drill-down)

Replaces list on mobile (same pattern as TaskDetailView, PatientDetail):

1. **Top bar**: ← 返回 (green) + "审核详情" title (48px, white bg)
2. **Patient header**: 44px avatar (rounded square) + name + age/gender + record type
3. **Structured field cards**: Each field in a tappable grey card (`#f7f7f7` bg,
   6px radius). Shows field label (11px, `#999`) + value (14px, `#333`) + ✏️ icon.
   Empty fields show "患者未提供" in italic `#ccc`.
4. **Interview chat history**: Collapsible card. Header: "▶ 问诊对话记录 (N轮)".
   Tap to expand full AI↔patient conversation.
5. **Sticky action bar**: Bottom-fixed, white bg, border-top. Two buttons:
   - "✓ 确认审核" — primary (`#07C160`, white text)
   - "✏️ 修改" — secondary (white bg, `#666` text, border)

### Inline editing

When doctor taps "✏️ 修改", the structured field cards become tappable.
Tapping a field:
1. Card background changes to white with a green left border
2. Value becomes a `TextField` (MUI) with the current value
3. Save/cancel buttons appear below the field
4. On save: PATCH API call, update local state, revert to read-only
5. Each edit creates a version entry (audit trail)

### Badge on Tasks nav icon

Combined count: pending tasks + pending reviews. Uses existing `Badge`
component with `color="error"`.

### Desktop behavior

Follows existing pattern: same content renders inside the Tasks tab area.
No split-view for P3 — that arrives with P2 when diagnosis results need
a side panel.

## Integration Points

### Interview pipeline → review queue

The integration point is `confirm_interview()` in
`src/domain/patients/interview_summary.py` (not `save_record()`). This
function currently:
1. Calls `save_record()` to create the `medical_records` row
2. Creates a `DoctorTask` of type `general` with title "审阅预问诊：{name}"
3. Sets `needs_review=True` on the medical record

**P3 change:** Replace the `DoctorTask` creation with a `review_queue` insert.
The existing task was a placeholder for the review workflow that P3 now
implements properly. Remove the `needs_review=True` flag (redundant with
`review_queue.status`).

```python
# In confirm_interview(), replace TaskRepository.create() with:
from db.models.review_queue import ReviewQueue
session.add(ReviewQueue(
    record_id=db_record.id, doctor_id=doctor_id,
    patient_id=patient_id, status="pending_review",
    created_at=_utcnow(),
))
```

The WeChat notification to the doctor is retained (best-effort, non-blocking).

### Review confirm → audit trail

Uses existing `medical_record_versions` table for field edits. The new
`old_structured` column captures the structured JSON before edits.
`audit()` helper logs `review.confirmed` and `review.field_edited` events.

### Chat history retrieval

Interview conversations are stored in `interview_sessions.conversation`
(JSON array column on `InterviewSessionDB`). The review detail endpoint
queries `interview_sessions` by `patient_id`, ordered by `updated_at DESC`,
and returns the `conversation` array from the most recent completed session
matching the record's time window.

## Success Criteria

- [ ] Interview completion auto-creates a review queue entry
- [ ] Doctor sees pending reviews in 待办 tab mixed with tasks
- [ ] Doctor can filter to 待审核 to see reviews only
- [ ] Tapping a review drills down to structured fields + chat history
- [ ] Doctor can inline-edit any structured field
- [ ] Doctor can confirm review with one tap
- [ ] Confirmed reviews visible in 已完成 for 2 days
- [ ] Every edit and confirmation logged to audit trail
- [ ] Badge count on Tasks nav reflects pending reviews
- [ ] Mobile UX matches existing TasksSection / PatientDetail patterns
