Task/Notification System Review

Codex (GPT-5.4) + Claude (Opus 4.6) independent analysis · 2026-04-13

Verdict: ~40% overbuilt, core is solid

The AI task extraction from records, date-grouped frontend, and notification delivery are well-designed. The overengineering is in infrastructure (distributed leases, 4-state lifecycle, 7 env vars, dead code) built for a multi-instance deployment that doesn't exist.

Both Agree

ALIGNED Drop notified as a status. Use a timestamp instead.

notified is a messaging event, not a task state. Every UI query already treats pending + notified identically. Replace with notified_at column. Two states: pending / completed (cancelled = soft delete).

ALIGNED 5 task types → 2-3 is enough.

No code branches on medication vs checkup vs follow_up — they share identical behavior. Ship with general (manual) + follow_up (AI-extracted). review already goes to the 审核 tab, not 任务.

ALIGNED Dead notification-preference code should be deleted.

src/domain/tasks/scheduler.py has regex parsers, Chinese commands, and 5 no-op async functions. The DoctorNotifyPreference table is already removed. This is dead code.

ALIGNED The distributed lease system is premature.

Full lock implementation with TTL, hostname:pid owners, fallback logic — for a single-instance product. The code even says "run anyway" when the lease fails. Delete SchedulerLease model + CRUD.

ALIGNED Keep a global task list, don't bury in patient detail only.

A doctor needs a cross-patient worklist. Per-patient task views are supplementary, not primary.

SIMPLIFY Scheduler: remove cron mode, keep interval only

Reduce from 7 env vars to 2: TASK_SCHEDULER_INTERVAL_MINUTES (default 10) and TASK_NOTIFY_RETRY_COUNT (default 3). Consider increasing interval from 1min to 10-30min.

SIMPLIFY Use due_on date instead of due_at datetime

Follow-ups are "check BP in 2 weeks" not "check BP at 14:37." Minute-precision creates false urgency. Date-only simplifies grouping and comparison.

Codex: "Use due_on date, not minute-precise due_at, unless a doctor explicitly sets a time."

Keep (well-designed)

KEEP LLM task extraction from records — Pydantic schema, max 5 per record, dedup index. Elegant.

KEEP Claim-before-send pattern — atomic WHERE clause for concurrent safety. Just change the implementation from status to timestamp.

KEEP Notification provider abstraction — log/wechat/wechat_mini_subscribe with fallback. Production-worthy.

KEEP Frontend date-grouped bucketing — overdue/today/tomorrow/this week/later. Good UX.

Defer (not needed yet)

DEFER Recurring tasks — "check BP every 2 weeks for 3 months." Real need, but implement when doctors ask.

DEFER Smart snooze/reschedule — "remind me again tomorrow." The due_at PATCH already enables manual rescheduling.

DEFER Merge 任务 tab into 我的AI — bigger UX change. Current separate tab works fine.

DEFER Per-doctor notification preferences — removing was correct. If needed later, add only coarse control: immediate / daily digest / quiet hours.

Missing Features Worth Noting

Codex's top picks

  1. Snooze/reschedule — without this, doctors ignore overdue or falsely mark done
  2. AI provenance — show source sentence + confidence, one-tap edit/dismiss
  3. Dedup/superseding — new visit notes create duplicate follow-ups
  4. Completion outcome — "BP stable, follow up in 1 month"

Claude's top picks

  1. Recurring tasks — "check BP every 2 weeks" has no model today
  2. Overdue-only WeChat push — don't push everything, only things the doctor missed
  3. Task dedup — LLM creates duplicates across records for same patient
  4. Source traceability — jump from task to the exact record/order that created it

Bottom Line

Codex

"You do not need a mini workflow engine for a solo specialist managing 10-50 patients. You need a trusted follow-up queue. Cut the model down to 1 core task type plus optional tags, open/done/cancelled states, due_on date, coarse notification mode, strong snooze, source traceability, and dedupe."


Claude

"The system is about 40% overbuilt for its current use case. The core value — 'AI extracts tasks from records, doctor sees them grouped by urgency, checks them off' — is solid and well-implemented. The overengineering is concentrated in notification delivery infrastructure built for a deployment that doesn't exist yet."
