# Design: P1 — Knowledge Base + Case History (Simplified)

> Generated by brainstorming session on 2026-03-20
> Status: APPROVED

## Goal

Add a case history knowledge base so that the P2 diagnosis pipeline can
reference similar past cases when generating differential diagnoses. Also
implement the skill loader so specialty-specific clinical knowledge (prompts,
diagnosis rules, clinical signals) can be loaded and injected into LLM calls.

## Context

Phase P1 of the clinical decision support roadmap. P3 (doctor dashboard +
review workflow) is shipped. P1 builds the knowledge foundation that P2
(diagnosis pipeline) will consume. P1 is backend-only — no UI changes.

This is the LLM-first / Path B approach: the LLM does clinical reasoning,
case history provides experience context, skill files provide domain knowledge.
No structured ontology, no question trees, no abstract interfaces.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Embedding model | BGE-M3 via sentence-transformers (local) | Best open-source Chinese medical embeddings. 100% local, no PHI risk. |
| Embedding provider | Abstraction layer with `EMBEDDING_PROVIDER` config | Local BGE-M3 now, Dashscope cloud documented as production alternative |
| Model loading | Startup preload | Predictable latency — no 5-10s delay on first query |
| Case creation | Auto on review confirm (skeleton) + doctor enriches later | Every interview becomes a case. Doctor promotes by adding diagnosis. |
| Confidence status | 2 values: preliminary / confirmed | Filter, not weight. Only confirmed cases participate in matching. |
| Seed data | Markdown file, doctor-readable | Neurosurgeon can review/edit clinical content. Same format as skills. |
| Diagnosis skill | Hybrid: output format + 10-15 must-not-miss patterns in Chinese | Enough to catch critical findings without encoding all of neurosurgery. |
| Skill loader | Full API per README spec, SkillType enum, 5-min cache | P2 needs it immediately. One implementation for all consumers. |
| Case matching | Cosine similarity, in-memory, confirmed cases only | <1,000 cases — no vector DB needed. Pure numpy operations. |

## Database Schema

### New table: `case_history`

```sql
CREATE TABLE case_history (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    doctor_id         VARCHAR NOT NULL,
    patient_id        INTEGER,
    record_id         INTEGER,
    chief_complaint   TEXT NOT NULL,
    present_illness   TEXT,           -- stored for embedding re-computation
    final_diagnosis   TEXT,
    key_symptoms      TEXT,           -- JSON array of strings
    treatment         TEXT,
    outcome           TEXT,
    notes             TEXT,
    confidence_status VARCHAR NOT NULL DEFAULT 'preliminary',
    embedding         TEXT,           -- JSON array of 1024 floats (BGE-M3)
    embedding_model   VARCHAR,        -- e.g. "BAAI/bge-m3" — tracks which model produced the vector
    created_at        DATETIME NOT NULL,
    updated_at        DATETIME,
    FOREIGN KEY (doctor_id) REFERENCES doctors(doctor_id),
    FOREIGN KEY (patient_id) REFERENCES patients(id),
    FOREIGN KEY (record_id) REFERENCES medical_records(id),
    UNIQUE (doctor_id, record_id)     -- one case per record per doctor (idempotency)
);

CREATE INDEX ix_case_history_doctor_confidence
    ON case_history (doctor_id, confidence_status);
```

### Confidence status

Two values only:

- `preliminary` — auto-created on review confirm. Has chief_complaint and
  present_illness from interview. No diagnosis, treatment, or outcome.
  **Excluded from case matching.**
- `confirmed` — doctor has added/verified the diagnosis. Full case with
  diagnosis + treatment + outcome. **Included in case matching.**

No `revised_by_id`, no `superseded` status. If the doctor corrects a case,
they edit it in place. Case history is a working knowledge base, not an
audit trail.

### Seed case ownership

Seed cases use a synthetic `doctor_id="__seed__"`. The `match_cases()` query
includes seeds alongside the doctor's own cases:
`WHERE doctor_id IN (?, '__seed__') AND confidence_status='confirmed'`.

This means all doctors see the same shared seed cases. When a doctor's own
confirmed cases accumulate, they naturally outweigh seeds in similarity
ranking. No per-doctor duplication of seed data.

The seed script checks existence before inserting (idempotent — safe to rerun).
The `UNIQUE (doctor_id, record_id)` constraint prevents duplicate cases from
production traffic. Seed cases have `record_id=NULL` (no linked medical record).

### Idempotency

- **Production cases:** `UNIQUE (doctor_id, record_id)` constraint. If
  `confirm_review_endpoint` retries or races, the second insert is a no-op
  (caught as IntegrityError, silently skipped).
- **Seed cases:** Script checks `WHERE doctor_id='__seed__' AND chief_complaint=?`
  before inserting. Safe to rerun without duplicates.

### Embedding storage

Embeddings stored as JSON array of 1024 floats in the `embedding` TEXT column.
BGE-M3 and Dashscope `text-embedding-v3` both produce 1024-dimensional vectors.
Same storage format regardless of provider.

**What gets embedded:**

- On create (preliminary): `chief_complaint + " " + present_illness`
  (both stored on the `case_history` row itself)
- On confirm: `chief_complaint + " " + present_illness + " 诊断：" + final_diagnosis + " 治疗：" + treatment`
  (all fields read from the `case_history` row — no need to re-fetch medical_records)

Re-embedding on confirm is important — the diagnosis text changes what the
case "means" semantically.

## Embedding Provider Abstraction

### Module: `src/domain/knowledge/embedding.py`

```python
EMBEDDING_PROVIDER = "local"  # or "dashscope"

# Interface
embed(text: str) -> List[float]                    # single text → 1024-d vector
embed_batch(texts: List[str]) -> List[List[float]]  # bulk embed

# Lifecycle
preload_embedding_model()  # called at app startup
```

### Local provider (default)

- Model: `BAAI/bge-m3` via `sentence-transformers` library
- GPU memory: ~2GB VRAM
- Speed: ~20ms per embedding (GPU), ~200ms (CPU)
- Preloaded at app startup alongside DB init and APScheduler
- Singleton: model loaded once, reused for all calls

### Dashscope provider (production alternative)

- Model: `text-embedding-v3` via `dashscope` SDK
- Dimensions: 1024 (matching BGE-M3)
- Speed: ~100ms per call (network)
- Cost: ¥0.0007 per 1,000 tokens (~free at private practice scale)
- preload is no-op (no local model)
- Requires: `DASHSCOPE_API_KEY` in config

### Config

All settings read from `config/runtime.json` (the sole config file per
AGENTS.md). Add a new `embedding` category to `config/runtime.json.sample`:

```json
{
  "embedding": {
    "EMBEDDING_PROVIDER": "local",
    "EMBEDDING_MODEL": "BAAI/bge-m3",
    "DASHSCOPE_API_KEY": "",
    "EMBEDDING_PRELOAD": "true"
  }
}
```

Read via `os.environ.get("EMBEDDING_PROVIDER", "local")` — the existing
`load_runtime_json()` flattens `runtime.json` categories into env vars.

**CPU/macOS fallback:** `sentence-transformers` automatically falls back to
CPU when no CUDA GPU is available (common on macOS dev machines). On Apple
Silicon, MPS backend may be used. First-call latency ~5s for model load,
~200ms per embedding on CPU. No explicit `device` config needed.

## Case History CRUD

### Module: `src/db/crud/case_history.py`

```python
async def create_case(
    session, doctor_id, record_id, patient_id,
    chief_complaint, present_illness,
) -> CaseHistory:
    """Auto-create on review confirm. Status=preliminary. Embedding computed
    from chief_complaint + present_illness. Both stored on the row."""

async def confirm_case(
    session, case_id, doctor_id,
    final_diagnosis, treatment=None, outcome=None, notes=None,
) -> CaseHistory:
    """Doctor enriches case. Status→confirmed. Embedding re-computed with diagnosis."""

async def update_case(
    session, case_id, doctor_id, **fields,
) -> CaseHistory:
    """Edit any field. Re-compute embedding if text fields changed."""

async def match_cases(
    session, doctor_id, query_text, limit=5, threshold=0.5,
) -> List[Dict]:
    """Find similar confirmed cases by cosine similarity.
    Filters: doctor_id match, confidence_status=confirmed.
    Returns top N with similarity score > threshold."""

async def list_cases(
    session, doctor_id, status=None, limit=50,
) -> List[CaseHistory]:
    """List cases for a doctor, optionally filtered by status."""
```

### Matching flow

```
New patient record
       │
       ▼
embed(chief_complaint + present_illness)
       │
       ▼
SELECT * FROM case_history
  WHERE doctor_id=? AND confidence_status='confirmed'
       │
       ▼
Load embeddings into numpy array (in-memory)
       │
       ▼
cosine_similarity(query_vec, case_vecs)
       │
       ▼
Return top 5 where score > 0.5
```

At <1,000 cases this is instant — pure numpy array operations, no vector DB.

## Skill Loader

### Module: `src/domain/knowledge/skill_loader.py`

Implements the API documented in `src/domain/knowledge/skills/README.md`.

### SkillType enum

```python
class SkillType(str, Enum):
    structuring      = "structuring"
    routing          = "routing"
    clinical_signals = "clinical_signals"
    diagnosis        = "diagnosis"
```

File mapping: `SkillType.diagnosis` → `skills/{specialty}/diagnosis.md`

### API

```python
def load_skills(specialty: str) -> Dict[SkillType, str]:
    """Read all .md files from skills/{specialty}/ + skills/_default/.
    Parse YAML frontmatter. Return {skill_type: content_body}.
    Cached per specialty, 5-min TTL."""

def get_skill(specialty: str, skill_type: SkillType) -> Optional[str]:
    """Get a single skill's content body."""

def get_structuring_skill(specialty: str) -> str:
    """Merge _default/structuring.md + {specialty}/structuring.md."""

def get_clinical_signals(specialty: str) -> Optional[str]:
    """Return {specialty}/clinical_signals.md content."""

def get_diagnosis_skill(specialty: str) -> Optional[str]:
    """Return {specialty}/diagnosis.md content."""

def get_routing_hints(specialty: str) -> Optional[str]:
    """Return {specialty}/routing_hints.md or _default/routing_hints.md."""

def list_specialties() -> List[str]:
    """List subdirectories of skills/ excluding _default."""

def invalidate_cache():
    """Clear the 5-min TTL cache."""
```

### Chinese specialty aliases

The README promises `load_skills("心内科")` works. The loader resolves
Chinese names to directory names via an alias map:

```python
_SPECIALTY_ALIASES = {
    "神经外科": "neurology",
    "神经内科": "neurology",
    "心内科": "cardiology",
    "心脏科": "cardiology",
}
```

`load_skills("心内科")` → resolves to `"cardiology"` → reads `skills/cardiology/`.
Unknown aliases pass through as-is (assumes English directory name).

### Cache

`_skill_cache: Dict[str, Tuple[Dict, float]]` — specialty → (skills dict,
timestamp). TTL configurable via `SKILLS_CACHE_TTL` env var (default: 300
seconds / 5 minutes). `invalidate_cache()` clears all entries.

### Frontmatter parsing

Skills use YAML frontmatter:
```yaml
---
name: unique-skill-name
description: one-line description
type: structuring | routing | clinical_signals | diagnosis
specialty: optional (defaults to directory name)
---
```

Body after frontmatter is the content injected into LLM prompts.

## Neurology Diagnosis Skill

### File: `src/domain/knowledge/skills/neurology/diagnosis.md`

```yaml
---
name: neurology-diagnosis
description: 神经外科鉴别诊断指导 — 输出格式与关键模式
type: diagnosis
specialty: neurology
---
```

**Content (~80 lines, Chinese):**

### Output format section
Defines the structure P2's diagnosis pipeline must produce:
- 鉴别诊断 (differentials): 诊断名称, 可能性 (低/中/高), 推理依据
- 检查建议 (workup): 检查名称, 理由, 紧急程度 (常规/紧急/急诊)
- 治疗方向 (treatment): 药物类别 (not specific doses), 干预方式
- 危险信号 (red flags): 需要立即处理的发现
- 免责声明: always "AI建议仅供参考，最终诊断由医生决定"

### Must-not-miss patterns (10-15)
- 突发剧烈头痛（"雷击样"）→ 首先排除蛛网膜下腔出血（SAH）
- 进行性头痛 + 视乳头水肿 → 颅内占位性病变
- GCS下降 + 瞳孔不等大 → 脑疝
- 马尾综合征（鞍区麻木 + 尿潴留）→ 急诊手术指征
- 4.5小时内急性肢体无力 + 言语障碍 → 脑卒中（溶栓窗口）
- 脊髓压迫（双侧肢体无力 + 感觉平面）→ 紧急MRI
- 局灶性起始的癫痫 + 结构性病变 → 考虑手术评估
- 三叉神经痛非典型特征 → 排除后颅窝占位
- 脑积水三联征（步态障碍 + 认知下降 + 尿失禁）→ 正常压力脑积水
- 外伤后清醒间期 → 硬膜外血肿

Medical abbreviations (SAH, GCS, NIHSS, MRI) preserved per AGENTS.md policy.

## Seed Data

### File: `data/seed_neurosurgery_cases.md`

Markdown format, doctor-readable/editable:

```markdown
# 神经外科种子病例库

## 头痛 — 脑膜瘤

**主诉：** 头痛反复发作2周，伴恶心呕吐
**现病史：** 前额持续性胀痛，晨起加重，伴恶心呕吐，近1周视物模糊
**诊断：** 脑膜瘤
**关键症状：** 头痛, 恶心, 视乳头水肿
**治疗：** 手术切除（开颅脑膜瘤切除术）
**转归：** 术后恢复良好

---

## 头痛 — 紧张性头痛
...
```

### Seed loader: `scripts/seed_cases.py`

Reads the markdown, parses `##` headers as case boundaries, `**field：**` as
key-value pairs (both full-width `：` and half-width `:` accepted, normalized).
Computes embeddings for each case. Inserts with `confidence_status=confirmed`.
Run once on setup.

**Field name mapping (Chinese markdown → case_history columns):**

| Markdown field | Column |
|----------------|--------|
| 主诉 | `chief_complaint` |
| 现病史 | `present_illness` |
| 诊断 | `final_diagnosis` |
| 关键症状 | `key_symptoms` (split by `,` or `，` → JSON array) |
| 治疗 | `treatment` |
| 转归 | `outcome` |

### 6 categories, ~5 cases each (20-30 total)

1. **头痛变体** — tension, migraine, intracranial mass/meningioma
2. **脑卒中/TIA** — ischemic stroke, hemorrhagic stroke, TIA
3. **脊髓压迫/椎间盘突出** — cervical/lumbar disc herniation, spinal tumor
4. **周围神经病变** — carpal tunnel, peripheral neuropathy
5. **癫痫/痫性发作** — focal seizure, generalized seizure, status epilepticus
6. **三叉神经痛/脑积水** — trigeminal neuralgia, normal pressure hydrocephalus

## P3 Integration

### Auto-create case on review confirm

When `confirm_review_endpoint()` sets `status=reviewed`, also create a
`case_history` entry. The `structured` dict must be fetched from the
underlying `medical_records` row (not available on the `ReviewQueue` object):

```python
# In confirm_review_endpoint(), AFTER db.commit() succeeds (best-effort):
try:
    from db.crud.case_history import create_case
    import json as _json

    async with AsyncSessionLocal() as db2:
        record = (await db2.execute(
            select(MedicalRecordDB).where(MedicalRecordDB.id == rq.record_id)
        )).scalar_one_or_none()
        if record and record.structured:
            structured = _json.loads(record.structured)
            cc = structured.get("chief_complaint", "")
            if cc:
                await create_case(
                    db2, doctor_id=resolved, record_id=rq.record_id,
                    patient_id=rq.patient_id,
                    chief_complaint=cc,
                    present_illness=structured.get("present_illness", ""),
                )
                await db2.commit()
except Exception as e:
    log(f"[review] case_history creation failed (non-blocking): {e}", level="warning")
```

**Best-effort:** Case creation runs in a separate session after the review
confirm succeeds. If embedding fails, DB errors, or structured fields are
empty, the review confirm is unaffected. The doctor's workflow never breaks
because of case history infrastructure. Failed case creations are logged
and can be retried manually or via a background reconciliation job.

## Case Enrichment Endpoint

P1 includes a minimal endpoint so the doctor can promote preliminary → confirmed
(without a full case management UI):

```
PATCH /api/manage/case-history/{case_id}?doctor_id=...
Body: { "final_diagnosis": "脑膜瘤", "treatment": "手术切除", "outcome": "术后恢复良好" }
```

Updates the case, sets `confidence_status=confirmed`, re-computes embedding
with diagnosis text included. Auth + rate limiting follows existing patterns.

This endpoint lives in a new `src/channels/web/ui/case_history_handlers.py`.
A full case management UI is deferred, but the API exists so the doctor
(or a future UI) can enrich cases.

## Deferred Items

1. **Case matching API endpoint** — P2 will expose matched cases through the
   diagnosis pipeline. No standalone API needed for P1.
2. **Doctor UI to manage cases** — full case list/edit/delete UI. P1 provides
   the PATCH endpoint; UI deferred.
3. **Embedding model hot-swap** — switching providers requires restart. Live
   swap deferred.
4. **Vector DB migration** — when case count exceeds ~1,000, migrate to pgvector
   or similar. In-memory cosine is sufficient until then.
5. **Multi-specialty seed data** — only neurosurgery seeds for P1. Cardiology
   and others when multi-specialty expansion happens.
