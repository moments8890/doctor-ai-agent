# Voice → Rule — Design Spec

**Date:** 2026-04-16
**Status:** Draft v4 (converged after 4 rounds of codex + Claude cross-review; codex 95% / Claude 94% implementability — pending user approval before implementation plan)

## Summary

Let a doctor add a clinical rule to their knowledge base by speaking a short
voice memo inside the WeChat miniapp. A new native miniapp page records the
memo and uploads to a new backend endpoint which transcribes it (Tencent
medical ASR) and LLM-extracts a candidate rule. The native page shows the
candidate inline for confirmation; confirmed rules land in
`doctor_knowledge_items` via the existing `POST /api/manage/knowledge`
write path with its existing `{content, category}` schema — no changes to the
write API or DB schema.

The goal is to lower the typing friction that gates knowledge-base activation.
Every other AI feature (diagnosis suggestions, reply drafts, red-flag
detection, rule citations) scales with KB richness, so lowering the cost to
add a rule compounds across the product.

## Goals

- First-time capture (mic tap → saved rule): 8–15s typical. Repeat captures
  should feel noticeably lighter than typing.
- Medical-term ASR accuracy is the ceiling available on this stack (Tencent
  `16k_zh_medical` against 16kHz mono MP3).
- End-of-speech to candidate visible: 3–8s on current batch path. Streaming
  transcript display (sub-3s perceived latency) is deferred — called out in
  non-goals and open questions.
- Extraction prompt iterates server-side without WeChat review cycles. The
  native page's layout and copy still require miniapp review; only
  server-controlled fields (rule text, error copy strings) iterate freely.

## Non-goals (v1)

- Multi-rule extraction from a single memo. v1 emits 0 or 1 rule; ambiguous
  multi-rule memos are rejected with a clear error, not silently collapsed.
- Voice for persona/style rules — target is clinical rules only
  (`doctor_knowledge_items`).
- Voice-triggered follow-up tasks, patient notes, or any capability other
  than clinical rule capture.
- Out-of-app capture via WeChat Official Account voice memo. Considered and
  rejected for v1 due to AMR narrowband accuracy loss and OA message
  round-trip latency. May revisit later as a complementary channel.
- **Streaming transcript display.** v1 is batch — doctor sees a spinner
  while ASR+LLM runs. Streaming (partial transcript visible as user speaks)
  is a perceived-latency win but requires a second endpoint + WebSocket
  wiring; deferred to v2.
- **Rich edit on the candidate card.** Doctor can either confirm as-extracted,
  edit the rule text in a plain textarea, or discard. No category editing in
  v1 (LLM picks, doctor accepts or re-records).
- **Deduplication against existing KB.** Current `save_knowledge_item`
  dedupes only on exact normalized text match. Voice paraphrase will bypass
  this and create near-duplicates. Flagged as known v1 risk with a v1.1
  mitigation proposal in Open Questions.
- **Automated miniapp UI tests.** Manual + real-device verification in v1.

## Key decisions (from brainstorm + review)

1. **Target data model: clinical rules only** — `doctor_knowledge_items`, not
   `doctor_personas`.
2. **Native miniapp page, not web-view or WeChat bot** — WeChat miniapp
   `<web-view>` cannot reliably access the microphone (persistent `getUserMedia`
   bug, confirmed in 2026 Chinese dev forums). Out-of-app WeChat bot loses
   ~20% medical ASR accuracy to AMR-NB 8kHz encoding and adds 5–10s of OA
   round-trip latency.
3. **Simple candidate schema matching existing write API** — candidate is
   `{content, category}`, matching what `POST /api/manage/knowledge` accepts
   today. No changes to the write path, no new DB columns, no confidence
   score in the client contract. Title/summary are auto-generated by the
   existing `save_knowledge_item` path, same as other add flows.
4. **New dedicated page `pages/add-rule`, not extending `pages/voice`** —
   cleaner single responsibility, no Swiss-army page growth. The existing
   `pages/voice/voice` (used for a different transcribe-and-return flow) is
   left alone.
5. **New endpoint `/api/manage/knowledge/voice-extract` bundles ASR + LLM** —
   one HTTP round-trip, one client-side state transition. Internally, the
   handler calls the same `services/asr/provider.py` module that
   `/api/transcribe` already uses (shared call site, not duplicated). It
   also mirrors `/api/transcribe`'s startup check that
   `get_asr_provider() != "browser"` — the endpoint fails fast in dev
   rather than silently returning empty transcripts.
   LLM extraction uses a new dedicated prompt file,
   `src/agent/prompts/voice_to_rule.md`, with a strict JSON output
   schema. The existing `knowledge_ingest.md` emits structured Chinese
   section headings (【主题】/【核心临床规则】/etc.) designed for
   document-scale OCR input; it is the wrong shape for a one-sentence
   voice memo. Inventing a dedicated prompt is the honest choice.
6. **1 memo = 1 rule for v1, but ambiguous multi-rule memos are rejected
   explicitly.** LLM is instructed to return `multi_rule_detected` error
   (not to silently pick one) when it sees more than one rule in the
   transcript.

## Architecture

```
┌─ Doctor webview (React SPA inside <web-view>) ────────────────────┐
│   AddKnowledgeSubpage — voice tab                                  │
│   Tap "🎙 语音添加" → openAddRuleVoice()                            │
│   → wx.miniProgram.navigateTo('/pages/add-rule/add-rule')         │
└─────────────────────────┬──────────────────────────────────────────┘
                          │
┌─────────────────────────▼──────────────────────────────────────────┐
│  Native miniapp page: pages/add-rule                               │
│  States: idle → recording → processing                             │
│          → candidate | error | perm_denied → saving → back         │
│  Recorder: wx.getRecorderManager (MP3 16kHz mono 96kbps)           │
│  Permission: wx.authorize({scope: "scope.record"}) on first use;   │
│              on denial → perm_denied state w/ wx.openSetting link  │
└─────────────────────────┬──────────────────────────────────────────┘
                          │ wx.uploadFile (multipart)
                          │   url: {apiBase}/api/manage/knowledge/voice-extract?doctor_id=...
                          │   header: Authorization: Bearer <token>
                          │   formData: (none beyond the file)
                          ▼
┌─────────────────────────────────────────────────────────────────────┐
│  POST /api/manage/knowledge/voice-extract                           │
│   Auth: _resolve_ui_doctor_id(doctor_id query, Authorization) —     │
│         same pattern as all /api/manage/* endpoints                 │
│   1. Delegate to services/asr/provider.py (SAME call site as        │
│      existing /api/transcribe — no duplication)                     │
│   2. Run LLM extraction (knowledge_ingest.md, possibly w/ voice     │
│      preamble). Outputs: {content, category, error?}                │
│   3. Return { transcript, candidate | null, error | null }          │
└─────────────────────────┬──────────────────────────────────────────┘
                          │
                          │ on confirm:
                          │ POST /api/manage/knowledge?doctor_id=X
                          │   body: {content, category} (EXISTING schema)
                          ▼
       doctor_knowledge_items row inserted via save_knowledge_item;
       idempotent on exact-text duplicate (returns existing row).
       wx.navigateBack; webview onShow refreshes KB list.
```

Properties:

- One native page, one new backend endpoint, one new prompt file
  (`voice_to_rule.md`), zero DB migrations, zero changes to existing
  write API.
- `pages/voice/voice` is not touched.
- `seed_source` column is NOT reused ad-hoc. If we want provenance ("this
  rule came from voice"), that's a separate follow-up — flagged in Open
  Questions, not shipped silently.

// miniappBridge.js
// Detection matches existing repo convention (see utils/env.js isMiniApp()):
// WeChat injects window.__wxjs_environment === "miniprogram" inside
// miniapp <web-view>.
export function isInMiniapp() {
  return typeof window !== "undefined"
    && window.__wxjs_environment === "miniprogram";
}

export function openAddRuleVoice({ onStaleVersion } = {}) {
  if (!isInMiniapp()) return;
  if (typeof window.wx?.miniProgram?.navigateTo !== "function") return;
  window.wx.miniProgram.navigateTo({
    url: "/pages/add-rule/add-rule",
    // `fail` fires when the target page isn't registered in this
    // user's installed miniapp version — catches the stale-cache case.
    fail: () => { onStaleVersion?.(); },
  });
}

## Backend contract

### `POST /api/manage/knowledge/voice-extract`

**Path namespace:** under `/api/manage/knowledge/` to match existing
sibling endpoints (`/process-text`, `/upload/extract`, `/fetch-url`).

**Request:**
- Query: `doctor_id=<id>` (same convention as all other `/api/manage/*`
  endpoints)
- Header: `Authorization: Bearer <token>`
- Body: `multipart/form-data` with an audio `file` field (MP3 expected;
  WAV/M4A tolerated — `services/asr/provider.py` runs ffmpeg transcode
  when needed)

**Auth:** handler calls `_resolve_ui_doctor_id(doctor_id, authorization)`
from `channels/web/doctor_dashboard/deps.py`, which delegates to
`resolve_doctor_id_from_auth_or_fallback`. Behavior: in production, valid
auth token required; in non-prod, the query-param `doctor_id` is accepted
as fallback (gated by `UI_ALLOW_QUERY_DOCTOR_ID` env flag, which is
already set in dev). This matches every other `/api/manage/*` endpoint.
**Miniapp-side**, `wx.uploadFile` supports both `header` (for
`Authorization: Bearer <token>`, confirmed in the WeChat miniapp API
docs — only `Referer` is forbidden) and URL query parameters, so both
pieces travel correctly. `wx.uploadFile` cannot send JSON bodies, but we
don't need one; the audio file is the only body.

**Response (success, rule extracted):**

```json
{
  "transcript": "前交通动脉瘤术后第二周要关注记忆问题",
  "candidate": {
    "content": "前交通动脉瘤术后第二周关注患者记忆变化，复诊时询问近期记忆清晰度",
    "category": "followup"
  },
  "error": null
}
```

`content` is a full-text clinical rule (what gets saved). Client renders it
as a card. Category enum → Chinese label mapping is hardcoded in the
native page (WXML can't import from React constants); pinned to:

| `category` | label | badge color |
|---|---|---|
| `custom` | 自定义 | gray |
| `diagnosis` | 诊断 | blue |
| `followup` | 随访 | green |
| `medication` | 用药 | orange |

These match `IconBadge` config in
`frontend/web/src/pages/doctor/constants.jsx`. If those ever drift, update
both sites.

**Response (handled errors — business-level):**

```json
{ "transcript": "...", "candidate": null, "error": "no_rule_found" }
{ "transcript": "",    "candidate": null, "error": "audio_unclear" }
{ "transcript": "...", "candidate": null, "error": "multi_rule_detected" }
{ "transcript": "",    "candidate": null, "error": "too_long" }
```

**Error policy:** business-level errors return HTTP 200 with `error` in the
body (so client has one response-handling branch regardless of outcome).
Operational errors (5xx, network, timeout) propagate normally so
observability and retry logic can distinguish. This is opinionated — docs
the decision explicitly.

### Save call (unchanged, existing endpoint)

```
POST /api/manage/knowledge?doctor_id=X
Authorization: Bearer <token>
Content-Type: application/json

{ "content": "<candidate.content>", "category": "<candidate.category>" }
```

Matches `AddKnowledgeRequest` in `knowledge_handlers.py:35` exactly. 3000-char
content cap already enforced server-side. `save_knowledge_item` handles
duplicates idempotently.

### LLM extraction

New prompt: `src/agent/prompts/voice_to_rule.md`.

**Strict JSON output contract** (enforced via LLM's response-format
control — request JSON-only, parse server-side with a fallback to
`error: "parse_failed"` on malformed output):

```json
{
  "content": "string | null",
  "category": "custom | diagnosis | followup | medication | null",
  "error": "null | no_rule_found | multi_rule_detected"
}
```

Success: `content` and `category` non-null, `error` null.
Handled failure: `content` null, `category` null, `error` is one of the
enum values. Handler translates this to the response-body `error` field.

**Prompt structure:**
- System/intro: "you receive a Chinese voice-memo transcript from a
  specialist doctor; filter out filler words and minor ASR noise; extract
  a single clinical rule; reject ambiguous multi-rule input"
- Few-shot: 4–6 pairs covering: clean rule per category, long-story-no-rule,
  multi-rule case, ambiguous-category case, memo with drug names
- Output directive: JSON only, no commentary

**Temperature:** 0.1. **Call path:** `src/agent/llm.py:llm_call(...,
json_mode=True)` which plumbs `response_format={"type": "json_object"}`
to the configured provider. The provider is whatever the project has
wired in `src/infra/llm/client.py` — a multi-provider registry
(DeepSeek/Qwen/SiliconFlow/Tencent-LKEAP), all OpenAI-compatible, all
with JSON-mode support. **The spec is not tied to a specific model; it
uses whatever the project default is at implementation time.** Preferred
approach: use the existing `structured_call()` helper with a Pydantic
response model so the JSON parse + fallback logic isn't re-implemented.

**Doctor specialty** (nullable in `Doctor.specialty`) is passed as
context if non-null; handler handles null gracefully by omitting it from
the prompt.

## Native page UI

### State machine

```
idle → recording → processing → candidate | error | perm_denied → saving → back
                                              ↑                       ↑
                                              └─── 重说 ────────────────┘

From perm_denied: [去设置] → wx.openSetting → idle (after user grants)
                  [取消]  → wx.navigateBack
```

### State details

| State | UI | Transitions |
|---|---|---|
| **idle** | Big centered mic button. Hint: *"长按说话，例如：前交通动脉瘤术后第二周要关注记忆问题"* | press (with permission granted) → recording; press (no permission) → `wx.authorize`; denied → perm_denied |
| **recording** | Mic pulses, live waveform OR simple dot pulse, elapsed timer, label *"松开结束"* | release → processing; swipe-up cancel → idle; hard cap at 60s auto-handoff to processing |
| **processing** | Centered spinner, label *"正在识别…"* | response 200 with candidate → candidate; response 200 with error → error; 15s timeout → error (network) |
| **candidate** | Card: `content` text (rendered as rule body), `category` badge (client-side label map). Buttons: **[保存]** primary, **[重说]** secondary, **[修改文字]** tertiary (opens textarea sheet where doctor can tweak `content` before save). Transcript accordion ("查看原话") at bottom. | 保存 → saving; 重说 → idle; 修改文字 → edit-sheet → candidate (modified content) |
| **saving** | Save button → spinner, disabled | POST success → `wx.navigateBack()`; POST failure → toast, stay on candidate |
| **error** | Friendly message (per error code) + **[重说]** + small **[取消]** | 重说 → idle; 取消 → navigateBack |
| **perm_denied** | Message: *"需要录音权限才能使用此功能"* + **[去设置]** + **[取消]** | 去设置 → `wx.openSetting`; 取消 → navigateBack |

### Error copy

| `error` value | Copy | Next |
|---|---|---|
| `audio_unclear` | *"没听清楚，请靠近麦克风再说一次"* | re-record |
| `no_rule_found` | *"没找到明确的规则。试试说：「当 X 时，要 Y」"* + show transcript | re-record or cancel |
| `multi_rule_detected` | *"听起来像多条规则，请一次说一条"* + show transcript | re-record or cancel |
| `too_long` | *"录音超过 1 分钟，请分条说明"* | re-record |
| (network / 5xx) | *"网络异常，请重试"* | retry |
| `rate_limited` | *"今日额度已用完，请明天再试"* | cancel only |

### Client-side limits & behavior

- Minimum recording: 1 second (anything shorter discarded client-side, no upload).
- Maximum recording: 60 seconds (client force-stops and hands off to processing).
- Upload retry: one automatic retry on network failure.
- Processing timeout: 15s from upload start to response.
- Rate limiting: **deferred to v1.1.** No per-doctor/per-day limiter
  currently exists in the codebase; adding one is non-trivial new infra
  (storage, reset-window logic). v1 relies on Tencent ASR's own provider-side
  quota as the upstream limit. The `rate_limited` error code is defined in
  the response schema so the client handles it gracefully if/when we add
  server-side limiting, but v1 handler does not emit it.
- `wx.onAppShow` / lifecycle: if the user backgrounds the app mid-recording,
  stop the recorder, discard the partial file, return to idle.

## Telemetry

Log on `/api/manage/knowledge/voice-extract`:
`{doctor_id, audio_duration_ms, transcript_length, endpoint_latency_ms,
error_code}`.

Log on save success:
`{doctor_id, candidate_edited: bool}`.

Log transcript text **only** behind a per-doctor opt-in flag (TBD where this
flag lives — per `doctor_profile` column or a separate settings table; the
plan decides). Transcripts may contain patient-identifying content; default
is off.

## Privacy / data flow

Voice audio and resulting transcripts traverse the same third-party providers
the product uses for text knowledge today:

- **Audio → Tencent Cloud ASR (`16k_zh_medical` engine).** Prod-path
  (Tencent) is in-memory only — the handler reads the uploaded bytes,
  base64-encodes for the Tencent API call, and discards. If the
  `whisper` provider is ever configured (dev/local), `provider.py`
  writes a tempfile that is deleted in a `finally` block. No audio
  persists beyond the request lifetime on our side.
- **Transcript → LLM provider** (per `src/infra/llm/client.py` registry)
  for extraction. **Honest disclosure:** `src/agent/llm.py` writes every
  LLM request (including transcript-as-prompt) to
  `logs/llm_calls.jsonl` on our backend. This logging is unconditional
  across all LLM-using features in the product today; voice does not
  introduce new persistence behavior, but the "opt-in transcript
  logging" in the Telemetry section applies only to *our* telemetry
  logs, not to the existing LLM call log. If PHI-compliance requires
  suppressing LLM call logs specifically for voice input, that is a
  cross-cutting change to `agent/llm.py` and is out of scope for this
  feature — flagged as a known gap, not silently assumed resolved.

Voice memos may contain more incidental patient identifiers (names, room
numbers, dates) than typed rules because speech is less filtered. Worth
noting in the doctor-facing first-use tooltip: *"录音仅用于识别规则，处
理完成后不保留原始语音"*.

No new third-party data flow is introduced by this feature; it extends
the existing data-flow envelope from "typed medical text" to "spoken
medical text." The doctor's general product-privacy consent covers this.

## Rollout

Ordering ensures the React button is never live before the miniapp page
is actually registered — avoiding the broken-`navigateTo` problem.

1. **Backend.** Land endpoint + `voice_to_rule.md` prompt + LLM eval
   suite + unit tests on `main`. Deploys freely, no WeChat review. Zero
   user-visible change (no client entry yet).
2. **Miniapp native page.** Land `pages/add-rule/`, register in
   `app.json`, submit miniapp release to WeChat review (3–7 days, longer
   for medical-adjacent scrutiny). Internal testers verify in WeChat
   DevTools simulator + real devices; `/api/manage/knowledge/voice-extract`
   is callable directly from native page for end-to-end validation.
3. **React bridge + button.** Only after miniapp release is approved and
   rolled out: delete dead voice scaffold in `AddKnowledgeSubpage.jsx`,
   add the new mic button + `miniappBridge.js`. Button is gated on
   `isInMiniapp()` so it's invisible in browser. **Stale-miniapp-version
   handling** (not all users update promptly): two mitigations layered:
   (a) add a `wx.getUpdateManager` call in the miniapp's `app.js` to
   force-refresh to the latest version on app launch — WeChat's default
   update cadence is not aggressive, so this is a real gap today;
   (b) wrap the `navigateTo` call in `openAddRuleVoice` with a `fail`
   handler that toasts *"请更新小程序到最新版本"* if the page isn't
   registered in that user's installed version. Both together close the
   window where a React-side button could dead-end on an older miniapp.
4. **Observe.** Watch telemetry: error-code distribution, save vs
   re-record vs discard rate, endpoint latency p50/p95.
5. **Iterate.** Weekly prompt / error-copy iteration during the first
   month based on observed extraction quality. No miniapp release
   required for these backend-only changes.

### Rollback

- **If the miniapp release is rejected or reveals a device-specific bug
  *before* step 3 ships:** React button doesn't exist yet. Fix forward,
  re-submit miniapp. No user impact.
- **If a bug surfaces *after* step 3:** revert the React commit that
  added the mic button (~5-minute hotfix). Backend endpoint becomes dead
  code (harmless). If miniapp itself needs changing, new submission →
  3–7 day review.
- **Fallback flag (optional, only if we want sub-minute disable):** gate
  the button render on a runtime config value (e.g., existing
  `config.ts` / Vite env). Only add this if telemetry shows we need
  faster-than-commit-revert control — likely overkill for v1.

## Open questions (deferred, with proposed handling)

- **Duplicate detection (v1.1).** `save_knowledge_item` dedupes only on
  exact normalized text; voice paraphrase will bypass it. Cheap v1.1 fix:
  on save, run a text-similarity check (e.g., simhash or TF-IDF cosine)
  against the doctor's last N rules in the same category; if >0.85
  similarity, show confirm sheet *"和已有规则「XX」很像，仍然新增？"*.
  Doesn't need LLM. ~1 day of work. Revisit once voice has a week of
  usage data.
- **Streaming transcript display** (sub-3s perceived latency). Requires a
  second endpoint (WebSocket or SSE) that returns partial ASR results as
  the user speaks. Defer to v2. Current 3–8s batch is usable but not
  delightful.
- **Low-confidence / ambiguous rules.** v1 contract doesn't carry
  confidence. If we find certain transcripts produce "technically valid
  but low-trust" rules (e.g., doctor spoke conjecture, not a rule), the
  LLM could emit a `low_confidence` flag and the card could render with a
  "请核对" treatment. Gate decision on first month of telemetry.
- **`seed_source` provenance.** Worth tagging voice-added rules as
  `seed_source="voice_memo"` so analytics can measure adoption. Not in v1
  until we've verified the column isn't consumed elsewhere in a way that
  would reject unexpected values.
- **Editing existing rules via voice.** Out of scope; v1 only creates.
- **Multi-rule memos.** Rejected in v1 with explicit error. If user
  research shows doctors consistently try to pack multiple rules, upgrade
  path is a prompt change + list-rendering card.
- **Rate-limit value.** 100/doctor/day is conservative guess. Telemetry-
  driven tune after launch.
