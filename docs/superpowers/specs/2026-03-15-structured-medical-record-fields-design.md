# Structured Medical Record Fields

## Goal

Persist the 13 regulatory-mandated outpatient record fields (per
《病历书写基本规范》卫医政发〔2010〕11号) alongside the existing free-text
narrative at record-creation time, eliminating the need for on-demand LLM
extraction during export and providing a stable, auditable, queryable
structured view of each medical record.

## Context & Motivation

### Regulatory basis

《病历书写基本规范》defines the medical record as a legal document created at a
specific point in time. Outpatient records must contain:

| # | Key | Chinese label |
|---|-----|---------------|
| 1 | `encounter_type` | 就诊类型 (初诊/复诊) |
| 2 | `department` | 科别 |
| 3 | `chief_complaint` | 主诉 |
| 4 | `present_illness` | 现病史 |
| 5 | `past_history` | 既往史 |
| 6 | `allergy_history` | 过敏史 |
| 7 | `personal_history` | 个人史 |
| 8 | `family_history` | 家族史 |
| 9 | `physical_exam` | 体格检查 |
| 10 | `aux_exam` | 辅助检查 |
| 11 | `diagnosis` | 初步诊断 (含 ICD 编码) |
| 12 | `treatment` | 治疗方案 |
| 13 | `followup` | 医嘱及随访 |

### Current state

- `MedicalRecordDB` stores `content` (free text) + `tags` (JSON keyword list).
- Structured fields exist **only transiently** during export, regenerated by an
  LLM call (`extract_outpatient_fields()`) each time.
- Problems: non-deterministic output across exports, requires live LLM for
  export, fields not queryable, audit trail is meaningless for a moving target.

### Design principles

1. **Legal determinism** — the structured record is created once and persisted.
   The same record always renders the same PDF.
2. **Narrative independence** — `content` (free text) and `structured_fields`
   (regulatory form) are two independent documents. Editing one does not
   auto-regenerate the other.
3. **Backward compatibility** — `structured_fields` is nullable. All legacy
   records (`NULL`) continue to work. Export falls back to LLM extraction.
4. **Single LLM call** — structuring prompt returns both narrative and 13 fields
   in one response. No extra latency or cost.

## Architecture

### Data flow

```
Chat input
    |
    v
_collect_clinical_text()          (unchanged)
    |
    v
structure_medical_record()        (prompt extended to return 13 fields)
    |
    +---> MedicalRecord.content           (free-text narrative — as today)
    +---> MedicalRecord.tags              (keyword list — as today)
    +---> MedicalRecord.structured_fields (NEW: dict of 13 regulatory fields)
    |
    v
save_record()                     (persists structured_fields to DB)
    |
    v
MedicalRecordDB                   (new column: structured_fields TEXT nullable)
    |
    +---> Export PDF -----> read structured_fields from DB (no LLM call)
    +---> Chat query -----> include in view_payload
    +---> Web UI ----------> display in record card / detail view
    +---> Patient portal --> excluded (privacy)
```

### 1. Structuring (`services/ai/structuring.py`)

#### Pydantic model (`db/models/medical_record.py`)

#### LLM prompt change

The structuring system prompt (stored in DB as `system_prompts.structuring`)
gains an additional instruction block requesting the 13 fields in the same
JSON response:

```
{
  "content": "...",
  "tags": [...],
  "record_type": "visit",
  "structured_fields": {
    "encounter_type": "初诊 或 复诊",
    "department": "科别（如：神经内科）",
    "chief_complaint": "主诉",
    "present_illness": "现病史",
    "past_history": "既往史",
    "allergy_history": "过敏史",
    "personal_history": "个人史",
    "family_history": "家族史",
    "physical_exam": "体格检查",
    "aux_exam": "辅助检查",
    "diagnosis": "初步诊断",
    "treatment": "治疗方案",
    "followup": "医嘱及随访"
  }
}
```

Rules for the LLM:
- Populate each field from the clinical content. Use `""` for fields with no
  available information (do not fabricate).
- `encounter_type` must be one of `"初诊"` or `"复诊"`.

#### Token budget change (`_make_llm_caller`)

Increase `max_tokens` from 1500 to 3000 in `_make_llm_caller()` at
`structuring.py:132` to accommodate the additional fields:

```python
max_tokens=int(os.environ.get("STRUCTURING_MAX_TOKENS", "3000")),
```

Making it env-configurable allows tuning without code changes.

**Partial field sets are expected and valid.** The LLM may return fewer than 13
fields if the clinical input doesn't contain information for all fields (e.g.,
a dictation that only mentions chief complaint and diagnosis). Empty string
values are acceptable.

#### Coercion (`_validate_and_coerce_fields`)

```python
from services.export.outpatient_report import OUTPATIENT_FIELD_KEYS

sf = data.get("structured_fields")
if sf is not None and not isinstance(sf, dict):
    data["structured_fields"] = None  # discard malformed
elif isinstance(sf, dict):
    # Ensure all values are strings, filter to canonical keys only
    data["structured_fields"] = {
        k: str(v) if v is not None else ""
        for k, v in sf.items()
        if k in OUTPATIENT_FIELD_KEYS
    }
```

### 5. Export — outpatient report (`outpatient_report.py`)

Replace the current "always LLM" flow with a DB-first approach:

```python
async def extract_outpatient_fields(records, patient=None, doctor_id=None):
    # Step 1: Try stored structured_fields
    stored = _merge_stored_fields(records)
    if _fields_sufficient(stored):
        log(f"[OutpatientReport] using stored fields doctor={doctor_id}")
        return stored

    # Step 2: Fallback to LLM extraction for legacy records
    log(f"[OutpatientReport] legacy fallback, calling LLM doctor={doctor_id}")
    return await _llm_extract_fields(records, patient, doctor_id)
```

**Sufficiency check**: `_fields_sufficient()` returns True if at least
`chief_complaint` and `diagnosis` are non-empty strings.

**Multi-record merge algorithm** (`_merge_stored_fields`):

```python
def _merge_stored_fields(records: list) -> dict[str, str]:
    """Merge structured_fields across records. Most recent record wins per field."""
    merged: dict[str, str] = {}
    # records are ordered DESC by created_at
    for rec in reversed(records):  # process oldest first so newest overwrites
        sf_raw = getattr(rec, "structured_fields", None)
        if not sf_raw:
            continue
        sf = json.loads(sf_raw) if isinstance(sf_raw, str) else sf_raw
        for k, v in sf.items():
            if k in OUTPATIENT_FIELD_KEYS and v:  # non-empty wins
                merged[k] = v
    return merged
```

**LLM fallback `max_tokens`**: The existing export extraction uses
`max_tokens=2000`. No change needed — this is only for legacy records and the
token budget is already adequate for extraction-only output (no narrative).

## Risks & open questions

1. **Token budget**: Increasing `max_tokens` from 1500 to 3000 doubles output
   cost per structuring call. Mitigation: the 13 fields are typically short
   (most are 1-2 sentences); real-world output should be 2000-2500 tokens.

2. **LLM field quality**: The LLM may produce sparse or inaccurate structured
   fields, especially for dictation-style input that lacks explicit history
   sections. Mitigation: empty fields are acceptable per the regulation; the
   doctor can edit via the web UI.

3. **Prompt backward compatibility**: The structuring prompt is stored in the
   `system_prompts` DB table and loaded at runtime. Updating it is a data
   change, not a code change. Must ensure the new prompt works with all
   configured LLM providers (DeepSeek, Ollama/Qwen, Claude, OpenAI).

4. **Consultation mode**: `structure_medical_record(consultation_mode=True)`
   uses a different prompt suffix. The structured fields instruction must
   work with both normal and consultation modes.
