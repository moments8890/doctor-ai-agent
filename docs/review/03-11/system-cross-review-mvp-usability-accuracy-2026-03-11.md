# 系统高层交叉评审：MVP / 易用性 / 准确率

## 结论

当前系统已经从“功能很多但主线不够清楚”的阶段，进入了“核心工作流开始成型，但实现和评估还没有完全收口”的阶段。

如果从 MVP、医生易用性、系统准确率三个维度一起看，当前最重要的判断是：

- 产品方向基本正确，尤其是 doctor-first、draft-first、explicit confirm 这条主线是对的
- Web 医生工作台已经接近可用 MVP
- 真正的主要风险不再是模型本身，而是范围过宽、跨渠道实现分叉、以及准确率评估流程仍然偏人工

一句话总结：

当前最该做的不是继续加新能力，而是把“一个医生主工作流 + 一套共享语义 + 一套可重复 benchmark”真正收紧。

## 最高优先级问题

### 1. MVP 边界仍然偏宽

从当前系统描述来看，产品同时承载了：

- Web 医生端
- 微信医生端
- 小程序 / 患者入口
- 任务、设置、后台类能力
- 多个 specialty / automation 方向

这会直接稀释 MVP 的清晰度，也会拖累易用性和准确率优化的节奏。

高层建议：

- MVP 只定义一个 hero loop：
  医生识别患者 -> 补充/生成病历草稿 -> 显式确认保存 -> 查询历史/继续随访
- 在产品层明确一个 primary channel
- 其余入口保留，但不要作为当前主评估对象

### 2. 准确率治理方向对了，但执行成本仍然过高

`e2e/README.md` 已经把 E2E 定义为系统准确率 gate，并明确 benchmark 必须跑在独立的 `8001` 服务上，这个方向是正确的。

但当前问题是：

- baseline / candidate 比较仍然依赖人工执行和人工判断
- 报告存在，但还没有收敛成一份稳定的、机器可读的结果摘要
- 如果 benchmark 过于依赖“人记得怎么跑”，它很容易在开发节奏中退化

高层建议：

- 把 benchmark 收敛成一条标准命令
- 输出统一格式的对比结果
- 明确 release 前必须比较的核心指标

否则“我们依赖 E2E 保证准确率”会停留在原则层，而不是工程现实。

### 3. 跨渠道 workflow 仍有分叉风险

当前文档已经明确希望 Web / WeChat 共享同一个医生工作流语义，但代码层仍然能看到不少渠道分叉的 orchestration。

这类问题的风险不是立刻报错，而是：

- Web 先修好
- WeChat 逻辑逐渐漂移
- 文档仍写“同一个 workflow”
- 最后产品体验和准确率在不同入口上慢慢不一致

高层建议：

- 把“患者绑定、pending draft、病历查询、确认写入”继续向共享 domain workflow 收口
- 渠道层尽量只负责输入输出适配，而不是各自维护业务语义

## 易用性评审

### 做对的部分

当前 Web 医生端已经明显更接近一个真实可用的医生助手，而不是一个泛后台：

- 默认入口是 chat / composer-first
- 页面有可见的 working context
- 当前患者、pending draft、next step 都能显式展示
- 草稿需要医生确认或放弃，不再 silently persist

这条线非常重要，因为医生场景里真正提升体验的，不是“页面很多”，而是“当前系统在帮我处理哪个患者、我下一步该做什么，一眼就知道”。

### 仍需收紧的部分

虽然 Web 方向基本对了，但 MVP 仍需继续克制：

- 不要过早把首页重新做成 dashboard / management UI
- 不要让 tasks / settings / admin 抢占默认认知焦点
- 如果一个医生第一次打开系统，最应该看到的是“现在可以开始处理患者”，不是“系统还有很多栏目”

另外一个低成本高收益点是 onboarding。

当前如果只采集医生姓名，那会错过一些对准确率和体验都很有价值的弱结构化信息，例如：

- specialty
- 常见就诊场景
- 常用病历风格

这些信息不需要变成复杂配置，但可以作为 routing / structuring 的 advisory context，性价比很高。

## 准确率评审

### 做对的部分

当前系统在“准确率”上的架构思路，整体是成熟的：

- authoritative context 和 advisory context 有明确分层
- 患者绑定、pending draft、确认写入不交给 LLM 自行决定
- `medical_records.content` 继续作为病历真相来源
- 草稿确认模型避免了“模型一旦误判就直接写库”的高风险路径

这说明系统已经理解：医疗场景里的准确率，首先是 workflow accuracy，不只是文本生成质量。

### 当前主要风险

当前对准确率的主要威胁，已经不是“模型不够强”，而是下面三类系统性问题：

#### 1. 上下文一致性

如果 patient binding、clarification、draft state 在不同路径里消费不一致，再强的 prompt 也无法稳定兜住。

#### 2. 评估闭环不够自动化

如果 E2E 跑法复杂、报告解读不统一，团队会更倾向于凭感觉判断改动是否“应该没问题”。

#### 3. 输入信号仍然偏薄

医生 profile、当前 encounter 类型、临床上下文过滤规则，都是低成本但高价值的准确率信号。当前合同已经有方向，但产品侧还没有把这些都吃满。

## 交叉评审判断

把架构、产品、前端、E2E 文档放在一起看，当前系统有三个明显特征：

### 1. 系统主线已经开始收敛

从最近文档和实现可以看出，团队已经在往以下主线收敛：

- 单一医生工作入口
- 显式 working context
- pending draft 保护
- cleaner LLM context assembly
- benchmark on separate deployment

这是好信号，说明系统不再主要依赖“功能堆叠”。

### 2. 当前最大的矛盾是“战略收敛”快于“实现收敛”

文档和原则已经比较清楚，但代码层仍然保留一些历史分叉和多入口负担。

所以现在最值得做的不是新增一轮大设计，而是把已有原则真正落实为：

- 更少的 workflow 变体
- 更少的渠道分叉
- 更少的 benchmark 操作自由度

### 3. 当前最强的产品特征是“安全感”，而不是“花哨能力”

这个系统最有价值的部分，不是它支持多少入口或多少 AI 功能，而是：

- 不会轻易把病历挂错患者
- 不会默默保存未确认草稿
- 医生知道系统当前在处理什么

这个方向应当继续强化。

## 建议的下一步优先级

### P0

1. 明确 MVP 的 primary channel 和 hero workflow
2. 把 benchmark 收敛成标准化对比流程，默认跑在 `8001`
3. 继续减少 Web / WeChat 在核心 doctor workflow 上的实现分叉

### P1

1. 在 onboarding 中补充最小医生画像信息
2. 把准确率指标从“跑过了”收敛到“哪些维度变好/变坏”
3. 把 clarification / patient continuity 进一步做成更稳定的 first-class context

### P2

1. 再考虑扩展更多 specialty 能力
2. 再考虑更复杂的管理端、患者端和自动化能力

## 最终判断

如果只看“系统已经有多少功能”，当前项目会显得很复杂。

但如果只看真正影响成败的东西，当前核心其实很简单：

- MVP 是否足够聚焦
- 医生是否一眼知道当前工作上下文
- 系统是否能稳定避免患者/病历错挂
- E2E benchmark 是否能真正成为准确率门槛

目前这四件事里，前两件已经接近成型，后两件还需要进一步工程化收口。

所以当前建议不是扩功能，而是收主线。
