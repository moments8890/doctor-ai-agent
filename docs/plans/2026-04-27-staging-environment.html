Staging Environment Plan

2026-04-27 · Codex 96% — SHIP_THIS · 17 tasks · ~half-day execution

TL;DR

What this plan ships

A parallel pre-prod stack on the same Tencent CVM. Push to gitee/staging auto-deploys to staging server. Tag v*-staging-* uploads the miniapp to WeChat 开发版. Real users on 正式版 see byte-identical behavior to today.

Why

Test WeChat miniapp + backend changes against real code without risking prod, before tagging a 正式版 release.

Risk profile

  • To public users: zero functional change. The miniapp's runtime resolver returns the prod URLs it returns today when envVersion === "release". Public WeChat users physically cannot reach 开发版 (WeChat platform-side gating).
  • To prod infra: non-zero. Same kernel, disk, MySQL instance, network. Minimized via cgroup caps, per-branch deploy locks, separate ports/schemas/paths. Residual risks (buffer-pool contention, instance outage, operator-error with wrong DSN) are explicit — escalate to dedicated CVM if any bite.

Critical surface — resolveBases() (Task 11)

The entire user-visible behavior change. ~15 lines. release branch returns the prod URLs unchanged from today.

function resolveBases() {
  // Devtools preview on a developer laptop: respect explicit config.js.
  try {
    if (wx.getSystemInfoSync().platform === "devtools" && runtimeConfig.apiBase) {
      return { webBase: runtimeConfig.apiBase, apiBase: runtimeConfig.apiBase };
    }
  } catch (_) { /* fall through */ }

  try {
    const env = wx.getAccountInfoSync().miniProgram.envVersion;
    if (env === "release") {
      return {
        webBase: "https://app.doctoragentai.cn",
        apiBase: "https://api.doctoragentai.cn",
      };
    }
    // develop / trial — only WeChat-listed members can reach this slot.
    return {
      webBase: "https://app.stg.doctoragentai.cn",
      apiBase: "https://api.stg.doctoragentai.cn",
    };
  } catch (_) {}

  return {
    webBase: "https://app.doctoragentai.cn",
    apiBase: "https://api.doctoragentai.cn",
  };
}
Why split webBase + apiBase? Because in prod, app.doctoragentai.cn has no /api/ proxy — it's static-only. The SPA fetches the API from api.doctoragentai.cn cross-origin. The miniprogram's wx.request calls hit a different host than the WebView src. Conflating them was Codex's #1 catch.

Critical surface — webhook rewrite (Task 3)

Per-branch dispatch + per-branch locks. main and staging deploys run in parallel; same-branch concurrent pushes are de-duplicated.

BRANCH_DEPLOYS: Dict[str, List[str]] = {
    "main": [
        "/bin/bash",
        os.environ.get("DEPLOY_SCRIPT_MAIN", "/home/ubuntu/deploy.sh"),
    ],
    "staging": [
        "/usr/bin/systemd-run",
        "--unit=staging-deploy-%s" % os.getpid(),
        "--slice=staging-build.slice",
        "--collect", "--wait", "--quiet",
        "/bin/bash",
        os.environ.get("DEPLOY_SCRIPT_STAGING", "/home/ubuntu/deploy-staging.sh"),
    ],
}

# One lock per branch.
_locks: Dict[str, threading.Lock] = {b: threading.Lock() for b in BRANCH_DEPLOYS}

The original webhook had a single global lock that dropped concurrent deploys. The naive ref-dispatch I first proposed had no lock at all. This version threads the needle: same-branch dedup, cross-branch parallel.

Critical surface — CI tag guards (Task 12)

Two workflows. Each gated by tag pattern (glob, not regex) and branch-of-tag verification.

WorkflowTag globJob-level filterBranch guardBuilds against
miniprogram-release.yml v* !contains(ref_name, '-staging-') tagged commit must be on main VITE_API_BASE_URL=https://api.doctoragentai.cn
miniprogram-staging.yml v*-staging-* (no extra filter — glob is precise) tagged commit must be on staging VITE_API_BASE_URL=https://api.stg.doctoragentai.cn
Failure modes the guards prevent: tagging a feature-branch commit as v1.2.3 (CI fails with "not on main"). Tagging a staging commit as v1.2.3 without -staging- (CI fails). Tagging a main commit as v1.2.3-staging-X (CI fails). Tagging anything else as v* (job is skipped, not failed).
